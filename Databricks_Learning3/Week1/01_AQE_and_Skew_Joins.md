# Part 3 Week 1①：AQEの魔法と、悪夢のData Skew（データ偏り）の克服

実稼働しているビッグデータ環境において、**「普段は5分で終わる夜間バッチが、今日はなぜか3時間経っても終わらず、最終的にエラー（OOM）で死んだ」**というトラブルが必ず発生します。
その最大の原因の99%が「Join（結合）時のデータの偏り（Data Skew）」です。これを見事に解決するのが、シニアデータエンジニアの最初の関門です。

## 1. 最強の自動救済機能：AQE (Adaptive Query Execution)
昔のSpark（バージョン2など）では、実行前に立てた「机上の空論の実行計画」のまま最後まで突っ走るため、途中でデータ量が爆発するとすぐに死んでいました。

現在のDatabricks環境（Spark 3以降）では、**AQE (Adaptive Query Execution：適応型クエリ実行)** がデフォルトで有効になっています。
これは**「途中でデータを読んでみたら予想より遥かにサイズが小さかったから、重いSort Merge Joinをやめて、爆速のBroadcast Hash Joinに作戦（物理計画）を変更するね！」**と、実行しながら自動で最適化してくれる魔法のような機能です。

## 2. AQEが助けてくれない「究極のData Skew」とは？
AQEには「Skew Join Optimization」という自動緩和機能もありますが、**極端すぎる偏り**に対しては無力です。

> [!WARNING]
> **🚨 実務での最悪のシナリオ（Data Skewの恐怖）**
> *   あるECサイトで「商品ID」をキーにして、10億件のトラフィックログとマスタ側をJoinさせます。
> *   しかし、そのサイトのトップページに表示されている期間限定の「0円キャンペーン商品（ID: 9999）」だけが異常なバズを起こし、ログの95%（9.5億レコード）が `id=9999` になっていました。
> 
> *   **どうなるか？**: Sparkは性質上、「同じIDを持つデータは、どれだけ巨大でも『1台のExecutor（ワーカーPC）』にまとめて送り込んでJOINさせる」という絶対ルールがあります。
> *   **結果**: クラスター内に100台のPCを用意していても、99台は数秒で仕事が終わって暇そうにしているのに、**残りのたった1台のPCが「9.5億レコードの処理」を丸抱えしてしまい、メモリがパンクしてクラスター全体がクラッシュ**します。（Straggler問題）。

## 3. シニアエンジアの奥義：「Salting (塩ふき) 技法」
AQEでも救えないこの状況に直面した時、コードを書き換えて手動で分散させる最終奥義が「**Salting（ソルティング）**」です。キーにランダムな「塩（ノイズ）」を振ることで、1台のPCへの一極集中を回避します。

### Step 1: 巨大データ側にランダムな「塩」を追加する
```python
import pyspark.sql.functions as F

# 問題の巨大データ (id=9999が9.5億個) に対して、"salt" という1〜10までのランダムな乱数列を新しく作って合成したキーを作る
huge_df_salted = huge_df.withColumn(
    "salted_id", 
    F.concat(F.col("id"), F.lit("_"), F.floor(F.rand() * 10)) # 結果: "9999_0", "9999_3" などバラバラになる
)
```
※これにより、`9999` というキーが強制的に10種類のバラバラのキーに分割され、Sparkは「別々のIDだ！」と勘違いして**10台のPCに綺麗に負荷を分散してくれます。**

### Step 2: 小さいマスタ側を「爆発（Explode）」させて塩に合わせる
しかし、マスタ側には `id=9999` の行は1つしかありません。「9999_3」といった偽のキーとぶつけるために、マスタ側も1〜10の連番をつけて意図的に10倍に増殖（Explode）させます。

```python
# 1〜10までの配列を作り、explodeして行を10倍に増幅させ、同じように合成キーを作る
master_df_exploded = master_df.withColumn("salt_array", F.array([F.lit(i) for i in range(10)])) \
                              .withColumn("salt_val", F.explode("salt_array")) \
                              .withColumn("salted_id", F.concat(F.col("id"), F.lit("_"), F.col("salt_val")))
```

### Step 3: 塩を振ったキー同士でJOINする！
```python
# 1台のPCがパンクすることなく、10台の別々のPCで並列にJoinが爆速で完了する！
final_df = huge_df_salted.join(master_df_exploded, "salted_id", "inner").drop("salted_id")
```

> [!TIP]
> AQEの自動最適化で解決できるうちは何もしないのがベストですが、「特定のNullや異常値のせいで何度やってもOOMが起きる」時の最終兵器として、このSaltingの概念を頭に入れておいてください。
