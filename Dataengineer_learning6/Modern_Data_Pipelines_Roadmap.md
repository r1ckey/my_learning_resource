# Part 6：モダン・データ配管工事の実戦 ロードマップ

「毎日、夜中の2時に前日のデータを全部（全量）読み込んで丸ごと上書き計算する（Full Refresh）」。
これは初心者として最初に書くべき素晴らしい設計ですが、データが「数億行」になった時、「昨日はたった1件しか売上が発生してないのに、今日も無駄に過去10年分の10億行を再計算して計算費用をドブに捨てるのか？」という大問題にぶち当たります。

**「新しく増えた部分（または更新された部分）だけを検知し、その【差分】だけを吸い上げて、元の巨大テーブルにマージ（Upsert）する」。**

この「増分処理（Incremental Processing）」こそが、実務（配管工事）でデータエンジニアが毎日死闘を繰り広げている領域です。
さらにこの章の後半では、パイプラインを作るオーケストレーター（Airflow）と変換器（dbt）における「やってはいけない実装アンチパターン（罠）」について解説します。

1. **Week 1: 究極の吸い上げ技術「CDC (Change Data Capture)」**
   * 「このデータ、昨日と何が変わった？」をSELECT文を使わずに検知する最強の技術。消された（DELETEされた）データも見逃さないDebeziumの仕組み。
2. **Week 2: 増分ロードの極意「ハイウォーターマーク（しおり）」**
   * 「前回どこまで読み込んだか」をどうやって記憶しておくか。Databricksの `Auto Loader` や `Structured Streaming` はこの仕組みをどう隠蔽しているのか？
3. **Week 3: Apache Airflow の深淵なる罠（アーキテクチャ）**
   * ジョブネットの制御で最も使われているAirflowにおいて、「タスク間でデータを渡す（XCom）」ことの危険性と、システムをクラッシュさせる待ち伏せ（Sensor）の罠。
4. **Week 4: dbt 黒魔術（Jinjaマクロによるメタ・プログラミング）**
   * 「50個の列を縦に並べ替える（Unpivot化する）長大なSQLを、どうやって『たった数行のForループ』で動的に自動生成させるか」という、ソフトウェア工学としてのSQLの極致。
