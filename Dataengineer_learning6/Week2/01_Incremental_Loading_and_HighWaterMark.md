# Deep Dive Week 2②：増分ロードの極意と「ハイウォーターマーク（しおり）」

前章のCDC（リアルタイム）を使わずとも、毎晩のバッチ処理や、ファイルシステム（S3やADLSにある毎日の売上CSVログ）からのロードにおいては、**「どこまで処理したかを記憶しておき、新しく増えた分だけ（増分）を読み込む」** 仕組みを絶対に設計しなければなりません。

これを手動で実装する際のアーキテクチャパターンと、クラウドがそれをどう自動化しているかの秘密です。

## 1. ハイウォーターマーク (High-Water Mark: 到達水位のしおり) 

「増分ロード（Incremental Load）」を設計する時、エンジニアは必ず「メタデータテーブル（記憶庫）」を一つ作ります。

*   **[ハイウォーターマーク（HWM）の概念]**:
    川の水が「どこまで水位が上がったか」を示す印のようなものです。本に挟む「しおり（Bookmark）」を想像してください。

1.  **初期状態**: 記憶庫 `last_load = 0` (まだ一回も読んでない)
2.  **初回ロードの実行**:
    *   システムは記憶庫を見に行きます（「よし、今回は ID `0` より大きいデータを全部取ってくるぞ」）
    *   ID `1〜1000` までの1000件のデータを引っ張ってきました。
    *   処理が終わった瞬間、システムは**記憶庫のしおりを `last_load = 1000` に更新・上書き保存（Commit・チェックポイント）**します。
3.  **翌日のバッチ実行（新しく10件増えていた場合）**:
    *   システムは記憶庫を見に行きます（「よし、前回は `1000` まで読んだな。今回は `1000` より大きいデータだけを取ってこい！」）
    *   ID `1001〜1010` までの「差分（増分）の10件」だけが完璧に抽出されます。

> [!WARNING]
> **🚨 破滅の罠（コミットするタイミング）**
> 面接でのキークエスチョン：「しおり（HWM）の数字を更新するのは、データを【抽出した直後】ですか？それとも【処理後の保存が成功した直後】ですか？」
> 
> **正解は「処理を終えて安全に保存（Commit）が行われた【完全な最後】」です。**
> もし抽出した直後に「1000まで読んだぜ！」としおりだけを更新してしまい、その後システムがエラーで止まった場合。
> 次に再実行した際、システムは「1000まで終わってるな、次は1001からだ！」と動き出し、**エラーで途切れた過去のデータ（1〜1000）が一切保存されずにブラックホールに消え去る（データの永久欠損）** という最悪の事態（非・冪等性）になります。

## 2. クラウドの魔法：Databricks Auto Loader と Structured Streaming

このように、手作りで「しおり（ハイウォーターマーク）」を管理し、エラー時のリトライなどを設計するのは、何百ものテーブルを管理するようになると非常に困難になります（State Managementの限界）。

それを完全に隠蔽・自動化してくれたのが、以前のパートで出てきた Databricks の **`Auto Loader`** や、Spark の **`Structured Streaming` (+ `checkpointLocation`)** という技術です。

*   **Databricksにおける魔法（Auto Loader等の裏側）:**
    あなたが `readStream().format("cloudFiles")` でS3バケットを監視させた瞬間、Databricksは裏側で**「RockDB」等を利用した強固な記憶領域（Checkpoint）を勝手に作成し、「どのファイル（どこまで）を読んだかのHWMのしおり」を、完全な耐障害性（失敗しても絶対に1ファイルも逃さず、かつ重複して読まない・Exactly-Once）を保ちながら全自動で書き込み・管理**してくれています。

データアーキテクト（あなた）は、あの「たった数行のAuto Loaderのコード」の裏で、このHWMと状態管理という血みどろの泥臭い複雑な処理が、数千人年のソフトウェアエンジニアの努力によって自動化されているという「裏の真実」を知る必要があります。
