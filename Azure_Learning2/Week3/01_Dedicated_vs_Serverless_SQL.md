# Azure Week 3①：Synapseの2つの頭脳「Dedicated」と「Serverless」

データ処理の最終着地点であり、経営層のダッシュボード（Power BI）に直結する最高峰のDWH（データウェアハウス）が **Azure Synapse Analytics** です。
DP-203試験において、このSynapseの内部にある **「2つの全く異なるSQLエンジンの使い分け」** で100%つまずきます。ここを完璧に設計できるかがアーキテクトの勝負です。

## 1. 専用 SQL プール (Dedicated SQL Pool)
昔は「SQL Data Warehouse」と呼ばれていた、ゴリゴリの強力なデータベース・サーバーです。（AWSのRedshiftに相当します）。

*   **特徴**: ADLSから一度**データを自分の中（専用のデータベース）にガッツリとインポート（取り込み）**してから、超高速で集計クエリを回す、重戦車のような存在です。
*   **課金体制（試験の肝）**: 「1000 DWU」といった性能の仮想マシン（ノード）をプロビジョニングするため、**クエリを全く叩いていない夜間でも「一時停止（Pause）」ボタンを押さない限り、毎月数十万円の莫大な維持費がダダ漏れになります**。

「夜間バッチが終わったら、必ず一時停止のスクリプトを組め！」というのが実務の絶対鉄則です。

### データの分散配置（Distribution）の恐怖
専用SQLプールは、複数台のサーバーにデータを千切って（分散して）保存します。
*   **ハッシュ (Hash) 分散**: 顧客IDなどのカラムをキーにしてデータを分散させる。**結合（Join）する相手のテーブルも同じキーで分散させておかないと、途端にクエリが数十分固まる（シャッフルエラー）**という落とし穴があります。
*   **ラウンドロビン (Round-robin)**: とりあえず順番にカードを配るようにランダムで千切る（ロードが爆速）。
*   **レプリケート (Replicate)**: 小さいマスタテーブル（「都道府県コード」等）を、全台のサーバーに丸ごとコピーして持たせておく（Hash結合の相手として最強に早い）。

## 2. サーバーレス SQL プール (Serverless SQL Pool)
現代のデータレイクハウス構築において、主力を担い始めている「もう一つの魔法のエンジン」です。

*   **特徴**: **「自分の中にデータベース（テーブル空間）を持たない」**という非常に珍しいSQLエンジンです。
*   **仕組み**: クエリ（SELECT）が走った瞬間だけ、背後に繋がっている **ADLS Gen2のデータレイク（CSVやParquet等）のファイルを直接生のまま覗きに行き、計算結果だけを返す**（Logical Data Warehouse）。

> [!CAUTION]
> **🚨 Exam Trap (コスト最適化の2択)**
> **[問題]**: 「ある企業では、毎月1回だけデータサイエンティストが、ADLSに保存されている数十GBの生ログ（Parquetファイル）に対して探索的なSQLクエリを叩いて調査したいと言っています。コストを最小限に抑えるにはどちらのSQLプールを使うべきか？」
> 
> *   ❌ 専用 SQL プール（データをインポートする手間と、サーバー作成・維持費がかかる）
> *   ⭕️ **サーバーレス SQL プール**
> 
> **[理由 (課金の仕組み)]**: サーバーレスSQLプールは「立ち上げ費用」や「維持費（一時停止）」といった概念が一切ありません。**『クエリがスキャンして調べたデータ量（1TBあたり約5ドル）』の従量課金**です。
> 「月に数回・気まぐれで生データをSQLで覗きたいだけ」なら、圧倒的に安く、環境構築が0秒で終わります。
