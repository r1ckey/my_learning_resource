# Part 5 Week 3①：自動化された入退社管理（SCIMとSSO）

数百人、数千人の社員が利用する大規模なデータプラットフォームにおいて、**「新入社員が来るたびに、情シス担当者が手動でDatabricksの管理画面を開いて、一人ずつ『Add User』ボタンを押してアカウントを作る」** という運用は1日で破綻します。
そしてこれが手作業で行われている場合、大企業のセキュリティ監査人から「退職した社員のアカウントが消し忘れられて残っていないか？」と指摘され、一発でアウトになります（重大インシデント）。

## 1. SSO (シングルサインオン) の基礎
Azure Databricks の最大の強みの一つは、**「Microsoft Entra ID (旧: Azure AD)」と完全に血が繋がっている（ネイティブ統合されている）こと** です。

AWS版等のDatabricksとは異なり、「Databricks専用のパスワード・画面」などは一切存在しません。
従業員は「いつもExcelやTeamsを開くときに使っている全社共通の顔パス（Entra ID）」の中にある「Databricks」というボタンを押すだけで、本人確認（多要素認証：MFAなど）の済んだ状態でそのままシームレスにログインします。

## 2. SCIM プロビジョニング (魔法の同期API)

手作業のユーザー管理を完全に自動化・撲滅する最強のエージェント技術が **SCIM (System for Cross-domain Identity Management)** です。

### ⚙️ 実務での最強のアーキテクチャ設計

1.  **Entra ID側でグループを作る:**
    大元のAzureポータル（Entra ID）上で、「Databricksを使うデータアナリストのグループ（`Grp-Data-Analysts`）」という枠（セキュリティグループ）を作ります。
2.  **エンタープライズ アプリケーション (SCIM) を設定する:**
    Entra IDの[エンタープライズ アプリケーション]という画面から、「`Grp-Data-Analysts` のリストに入った人間やグループを、全自動でDatabricksの中に投げ込み（流し込み）続けろ」という命令（Provisioning）を設定します。
3.  **SCIM同期の「40分」の魔法:**
    約40分おきに、Entra IDがDatabricksに向けて定期的に通信を行います。「このAさんは新しくグループに入ったからアカウントを作れ」「Bさんは部署を異動になってグループから外れたから、Databricksの中から即座にアカウントを消せ（無効化しろ）」。

> [!IMPORTANT]
> **🚀 アーキテクトが見据える「完全な自動化」**
> あなた（データ基盤のアーキテクト）の仕事は、たったこれで終わりです。
> 
> これ以降、新入社員の入社処理や退職者の削除は**「人事部」や「ITサポート部門」の仕事**になります。彼らがいつも通りに社内のEntra IDの人事マスタを更新するだけで、裏側でSCIMが自動的に走り、**あなたの預かり知らぬところで、Databricks上に勝手にアカウントが作られ、勝手に適切な権限が与えられ、退社日には勝手にアクセス権が消滅して締め出されます。**
> 
> 「属人化」と「システム管理者の過労死」を防ぐ、非常に美しいゼロタッチ・デプロイメントの完成です！
