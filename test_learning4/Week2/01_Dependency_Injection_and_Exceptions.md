# Test Week 2④：例外（Exception）の人工発生と「DI (依存性の注入)」

前章の「スタブ（Stub）」。これを使うと、プログラマの念願だった**「数万回のテストに1回しか起こらないような、奇跡の異常系エラー（Unhappy Path）」をローカルのPC内で100%確実に、意図的に引き起こす**ことができます。

## 1. 異常系のシミュレーション（意図的な爆破）

**【自分の書いたソースコード (Python)】**
```python
def fetch_user_data(user_id):
    try:
        data = database.get_user(user_id) # ←DBから取得
        return data
    except TimeoutError:
        return "ただいま回線が混雑しています" # ←ここをテストしたい！！
```

この `except TimeoutError:` のルート（異常系）を通すために、テスト中にわざわざ家のルーターのLANケーブルを物理的に引っこ抜いてDBを「5000秒タイムアウト」させるのは非効率です（自動化できません）。

そこで、前章の「スタブ」に悪意のある指示を出します。
**「おい、データベースの影武者（スタブ）君。君は今から `get_user`と呼び出された瞬間、わざと『TimeoutError』という爆弾（Exception）を投げて自爆してくれ！」**

多くのテストフレームワーク（Pythonの `unittest.mock` や `pytest`、Javaの `Mockito`）には、モック・スタブに「例外をRaise（送出）させる」機能が付いています。
これにより、プログラマーは「いつでも好きな時に、好きな種類のエラーを人工的に発生させ、自分のコードが安全にエラーメッセージを返して（ソフトランディングして）くれるか」を秒速で自動テストできるようになります（異常系の完全制覇）。

---

## 2. アーキテクチャの壁「DI (Dependency Injection: 依存性の注入)」

しかし、上記のように「部品を影武者（スタブ・モック）にすり替える」ためには、あなたの書いた**ソースコード自体が、【すり替え可能（部品を交換可能）な『DI』の設計】で書かれていなければならない**という壁にぶち当たります。

### ❌ テスト不可能な最悪のコード (密結合 / 依存している状態)
```python
def fetch_user_data(user_id):
    # 【最悪】関数の『中』で、本物のDB接続部品（Oracle）をガッチリ固定で生成（密結合）してしまっている！
    db = RealOracleDatabaseConnection(...) 
    
    return db.get_user(user_id)
```
このコードは、関数の「中」で本物のDBとズブズブに癒着しているため、外側からテストの時に「影武者に入れ替える」隙が1ミリもありません（テスト不可能なクソコードです）。

### ◯ DI (依存性の注入) された美しいコード (疎結合)
```python
# 【最高】DB部品を「関数の引数（外からの注入口）」として受け取るように設計（DI）している！
def fetch_user_data(user_id, database_component):
    return database_component.get_user(user_id)
```

この疎結合（緩い作り）にしてあるコード（DI設計）であれば、以下の通り**状況に応じて「本物」と「影武者」を自由に差し替えて注入（Injection）**できます！

*   **[本番環境（アプリ起動時）の場合]**:
    `fetch_user_data( 1, 本物のOracle_DB部品 )` （本番は本物に接続）
*   **[テスト環境（UT実行時）の場合]**:
    `fetch_user_data( 1, わざと自爆するスタブ君 )` （テストの時は偽物を流し込む！）

> [!IMPORTANT]
> **🚀 「テストしやすいコード」＝「変更に強い良質なアーキテクチャ」**
> プログラマーが『単体テスト（UT）用のコード』を書く最大のメリットは「バグを見つけること」だけではありません。
> 「自分の書いた関数が、モックにすり替えられない（テストコードが書きづらい！）」と気づいた瞬間、そこには必ず**「オブジェクト同士が密結合しすぎている、将来変更に弱い粗悪なアーキテクチャ（設計）」**が潜んでいます。
> テスト（TDDなどの思想）は、ソースコードの品質そのものを強制的にDI（疎結合）の美しい姿へと矯正・洗練させてくれる最高のリファクタリング・ツールなのです。
