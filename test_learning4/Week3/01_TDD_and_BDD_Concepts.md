# Test Week 3④：開発パラダイムの反転「TDD と BDD」

「システム（ソースコード）を作り終わってから、最後にそれが動くかどうかテストするための（確認用）コードを書く」。
これが一般人の常識でした。しかし、アジャイル開発の世界では、この順序を**180度完全に逆転させる**という狂気の（そして最も美しい）設計思想が生まれました。

## 1. TDD (Test-Driven Development: テスト駆動開発)

**「コードを書くより前に、まず【空っぽのテストコード】から書き始めろ」**

これがTDDの思想であり、開発の進め方を「テストコードを主軸（駆動エンジン）」にして回していく手法です。
有名な3つのサイクル（**Red ➡️ Green ➡️ Refactor**）を無限に繰り返します。

1.  **🔴 Red (失敗するテストを書く)**:
    「足し算関数に『1』と『2』を入れたら『3』が返ってくること」という【期待値の要件（テストコード）】だけを先に書きます。もちろん、本体の足し算関数すらまだ存在しないので、テストを実行すると無惨に「エラー（Red：失敗）」になります。
2.  **🟢 Green (どうにでもいいから通す)**:
    「エラーを消して、テストコードを合格（Green：成功）」させるためだけに、最短最速の（ダサい）プロダクトコードを書きます。
    例えば `def add(a, b): return 3` のように、無理やり固定値の `3` を返す嘘のコードでも構いません。まずはテストを通過（Green）させます。
3.  **✨ Refactor (美しいコードに洗練する)**:
    テストがGreenになっている（仕様を満たしているという後ろ盾の安心感が100%ある）状態で、先ほどのダサい `return 3` の関数の中身を、本格的な `return a + b` などの美しいアルゴリズムへと「リファクタリング（再構築）」していきます。編集中に誤ってコードを壊しても、瞬時にさっきのテストコードが（Red）で知らせてくれるので、絶対にバグを作り込みません。

常に「今やるべき要件（テスト）」が先にあり、それを「実装」が後から追いかける。これにより「使われない無駄なコード（オーバースペック）」が1行も書かれなくなり、コードは常に100%テストで保護された状態になります。

## 2. BDD (Behavior-Driven Development: 振る舞い駆動開発)

TDDは素晴らしいですが、少しばかり「プログラマー（技術者）視点すぎる」という欠点がありました。（例：この関数は引数が配列で〜という技術的なテストになりがち）。

そこで、「顧客（ユーザー）がどういう行動（振る舞い）をした時に、アプリがどういうビジネス的価値（動作）を返すべきか？」という**【ビジネスの要件定義（言葉）】**と、【テストのプログラム・コード】を完全に一体化させるアプローチが開発されました。それが **BDD（振る舞い駆動）** です。

有名な Cucumber などのツールを使い、**『Gherkin（ガーキン）構文』**という素人（英語/日本語）でも読めるフォーマットで要件定義書を書きます。

```gherkin
// ユーザーとビジネス側が一緒に書く「要件定義書（振る舞いのストーリー）」
Feature: ネットスーパーの割引キャンペーン

  Scenario: プレミアム会員は50%オフになること
    Given  ユーザーは「プレミアム会員」である  // （前提条件: Given）
    When   彼らが「カート画面」に移動した時    // （行動操作: When）
    Then   合計金額の表示は「50%オフ」になっているべきである // （期待値: Then）
```

驚くべきことに、BDDのツール（フレームワーク）は、**「この日本語（英語）で書かれた全くプログラミング言語ではない要件定義書のテキスト」自体を、そのまま「テストの自動実行スクリプト」として強引に（裏でPythonやJavascriptのマクロと紐付けて）自動テストに組み込んでしまいます！**

> [!TIP]
> **🌟 ドキュメントの陳腐化を防ぐ（生きた設計書）**
> 通常、ExcelやWordで書いた仕様書は、プログラムが改修されるとメンテされなくなり、いつか嘘のこと（バグ）が書かれた「古い使えない紙切れ」になります。
> しかしBDDスタイルであれば、**『この日本語の文章（仕様書）自体が、テストコードそのものとして動いている』**ため、仕様書通りにアプリが動かなければテストがエラーになり即座にエンジニアが気付きます。
> これにより、「絶対に嘘をつかない（アプリと100%同期し続ける）、生きた要件定義書（Living Documentation）」を作り出すことができるのです！
