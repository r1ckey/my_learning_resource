# Part 4：単体テストの実装アーキテクチャ・モック活用 ロードマップ

ここまでExcelで「異常系のテスト（例：DBの電源が落ちたら）」といったテストケースを人間向けに作ってきました。
しかし、いざプログラマーがそれを自分のPC（PythonやJavaのコード）で自動テスト（Unit Test / UT）として組み込もうとした時。
**「自分のローカルPCの中で、テスト中・関数実行中の『1ミリ秒の間』に、どうやって狙ってOracleデータベースの電源を引っこ抜いて（DB通信切断エラーを起して）、異常系プログラムが動くかテストするんだよ！物理的に不可能だろ！」**と発狂します。

ここで、テスト業界の最大の魔法である**「テスト・ダブル（影武者・スタブ・モック）」**という概念が登場します。

1. **Week 1: 代役を立てる「テストダブル (Test Doubles)」**
   * 「この機能はまだ完成していないから」「本物のDBに繋ぐとテストが遅くなるから」といった理由で、本物のシステムパーツを【偽物のモジュール（影武者）】にすり替えてテストを続行する技術（Stub/Mock/Spy等の使い分け）。
2. **Week 2: 依存性の排除 (DI) と「例外の人工発生」**
   * プログラマブルに「DBが通信タイムアウトしたフリ（ExceptionのRaise）」を人工的に起こさせる方法。またそのためには、コードが「DI（依存性の注入）」という綺麗な設計で書かれていなければならない絶対的理由。
3. **Week 3: TDD（テスト駆動開発）とBDD（振る舞い駆動開発）**
   * 「コードを書いてから、それが動くかテストを書く」というこれまでの常識を180度反転させ、「まず空っぽのテストから書いて、それを満たすようにコードを書く」レッド・グリーン・リファクタリングの芸術。
4. **Week 4: データエンジニアリングにおける「品質保証」の姿**
   * アプリ開発（画面ボタン等）から離れ、私たちデータエンジニアの主戦場である「パイプライン・データウェアハウス」において、バグ（不正データ）の侵入を監視する「dbt」等のテストエコシステムへの回帰。
