# Test Week 2③：「期待値」の掟（曖昧な日本語の禁止）

Excelのテスト仕様書の横軸（カラム）は、一般的に以下のような列で構成されます。

1. **テストID**: `UT-LOG-001` のようなユニークな番号。
2. **事前条件**: 「DBにユーザー『テスト太郎(ID:1)』が登録済であること」など。
3. **テスト手順**: 「1. ユーザーIDプルダウンから『テスト太郎』を選択。2. 『削除』ボタンを押す」
4. **【期待する結果 (Expected Result)】**: ここが仕様書における最も重要な列です。

## 1. 初心者の罪「正しく〜されること」

テスト初心者が陥る「仕様書を全く使い物にならなくする」2大・曖昧ワードがこれです。

*   **「正しく（問題なく）表示されること」**
*   **「エラーになること」**

**[なぜこれが罪なのか？]**
もし、実行者（アルバイトのテスター等）がこの文章を読んで、画面が真っ赤になって「`Fatal Error 500: Database Broken`」と表示された場合。テスターは「あ、（システム的には）エラーになったな。よし、期待値通りだ！【テスト合格（OK）】！」とハンコを押してしまいます。
本来そこには「赤字で『IDが不正です』という親切なメッセージが出て、ログイン画面に引き返す」というUIが期待されていたにも関わらず、**「曖昧な期待値のせいで、致命的なバグが『OK（合格）』として見逃された（出荷された）」**のです。

## 2. 厳密な「期待値」の設計（事実と数値の記載）

プロのQAアーキテクトが書く・レビューする「期待値」のカラムには、テスターの【主観や解釈】が入り込む余地を1ミリも残しません（誰が実行しても同じ結果の判定になる）。

### 🌟 良い期待値の書き方の具体例

*   **UI（画面）の場合:**
    「エラーメッセージポップアップが開き、中に『パスワードは8文字以上で入力してください』というテキストが赤字（#FF0000）で表示されること。かつ、元の入力フォームのPW欄のみが空欄にリセットされていること（IDは残っていること）。」
*   **データベース（バックエンド・API）の場合:**
    「HTTPステータス `200 OK` が返却されること。さらに、バックエンドのデータベースの `users` テーブルにおいて、`userId = 1` のレコードの `status` カラムが `'active'` から `'deleted'` にUPDATEされていること。かつ、本当に物理削除されていない（レコードが消えていない）こと。」

> [!IMPORTANT]
> **🌟 Evidence（エビデンス：証拠のスクショ）の定義**
> 
> 期待値が曖昧でなければ、「何を証拠（エビデンス）として提出すればいいのか」も自ずと明確になります。（例：「このDBの一行が変わったところのSQLのスクショを貼れ」）。
> 
> 期待値の列を書く時、常に自分に問いかけてください。**「もし見ず知らずの他人が読んで実行した時、この1行で、どうやって『真実のバカルール（合格か否か）』を判定できるか？」**。
> 「正しく動くこと」を禁止し、現象（Fact）のみを記述する。これがテストの世界の掟です。
