# Part 3：Excelテスト仕様書の実務作成術 ロードマップ

「同値分割」や「デシジョンテーブル」を使って、テストすべき「点（観点）」は完璧に抜き出すことができました。
しかし、それを**「Excel（またはスプレッドシートやJira等）」のフォーマット（行と列の形）に整然と書き写せなければ、100人規模の開発プロジェクト（日本のSIerやグローバルチーム）は機能しません。**

現場に配属されて最初に任される仕事でありながら、最も「センスの無さ（ぐちゃぐちゃの書き方）」が露呈し、他人がテストできずにプロジェクトが破綻する原因となるのが**【ドキュメンテーション（仕様書作成）のスキル】**です。

1. **Week 1: 香盤と階層構造（大・中・小項目の切り出し）**
   * Excelで数十万行もあるプロジェクト。行を「大項目」「中項目」「小項目」にどうやってブレイクダウンしていくか。テストを無駄なく効率よく進めるための「シナリオ」の組み立て方。
2. **Week 2: 「期待値」の書き方の掟（絶対的ルール）**
   * 「画面が正しく表示されること」「エラーになること」。素人がよく書くこのフワッとした日本語の「期待値（Expected Result）」が、プロジェクト全体を破壊する理由。
3. **Week 3: 組み合わせ爆発の回避（Pairwise法と直交表）**
   * デシジョンテーブルを使っても、「OS3種類 × ブラウザ4種類 × ユーザー権限3種類 × 割引5種類 ＝ 180通り」と、現実的にテスト不可能な膨大な数にぶち当たった時。これを「たった十数件」に圧縮しても品質を保証できる数学の魔法。
4. **Week 4: テストのレビューと「やる意味のないテスト」の排除**
   * 書かれたテスト仕様書の中で、「それはただのフレームワーク（SQLやAzure自身）のアタリマエの挙動をテストしているだけだろ！」という、お金と時間の無駄（テストのためのテスト）を削ぎ落とすアーキテクトの視点。
