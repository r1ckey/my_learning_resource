# Part 5 Week 2①：NPIPとセキュアクラスター接続 (SCC)

大企業のインフラ担当者（AZ-104所持者）とDatabricks導入のミーティングを行うと、彼らは必ず鬼の形相でこう言います。
**「君が動かそうとしているDatabricksの仮想マシン（Sparkノード）に、『Public IP（グローバルIPアドレス）』は付与されるのか！？ 1つでも付いているなら、セキュリティ監査違反で導入はさせられない！！」**

これに対する完璧な防御壁となるのが、**No Public IP (NPIP)** または **セキュアクラスター接続 (Secure Cluster Connectivity: SCC)** と呼ばれるアーキテクチャです。

## 1. 昔のDatabricksの危険な挙動
数年前までのDatabricksでは、自社のVNet内（データプレーン）に立ち上がったSparkクラスターのVMの1つ1つに、インターネットからアクセス可能な**「Public IPアドレス」がむき出しで付与**されていました。

なぜでしょうか？
それはVMたち（計算奴隷）が、遠隔にいるコントロールプレーン（司令塔）とインターネット越しに通信して、「今日の仕事（Notebookのコード）をください！」と言ってコードを受け取ったり、結果を報告したりするために、外に出るためのIPが必要だと思われていたからです。

*   **インフラ担当の怒り:** 「Public IPが付いているということは、世界中のハッカーから直接アクセス（SSH等）を受け得る的（マト）になっているということだ。絶対に許さん！」

## 2. NPIP による「出口の封鎖」

ワークスペースを作成する際の最後のタブに、**`[ ] パブリック IP なし (No Public IP)`** または **`[ ] セキュアクラスター接続を有効にする`** という小さなチェックボックスがあります。（※現在は大半のリージョンでデフォルトONになっています）。

> [!IMPORTANT]
> **🌟 Secure Cluster Connectivity (SCC) の魔法の動き**
> もしクラスター側のVMにパブリックIP（出口経路）がないなら、どうやって外の世界にいる司令塔（コントロールプレーン）と連絡を取り、命令を受け取るのでしょうか？
> 
> **[仕組みの核心]**
> SCCがONになると、私たちのVNet（データプレーン）の中に、Azure全体を繋ぐ巨大な見えない中継器 **『コントロールプレーン・リレー（Control Plane Relay）』** へのトンネルが形成されます。
> *   Spark側のVM（完全なプライベートIP＝引きこもり）から、「内側から外側へ」のインターネットを通らない安全な通信（アウトバウンド）だけを使って、この中継リレーに対して常に通信接続（トンネル）を張りっぱなしにします。
> *   司令塔（コントロールプレーン）は、直接VMに命令を送るのではなく、この安全な中継リレーからトンネル（既に繋がっている糸電話）を通じて、命令を内部に流し込みます。
> 
> これにより、VM側は世界への窓（Public IP）を完全にコンクリートで塞ぎながら、安全に命令を受信できるようになりました。

## 3. 外の世界（SaaSやAPI）と通信したい場合

「Public IPを塞いだのはいいけど、じゃあ『外部のAPI（Slackへの通知）や、Azure以外のSaaS（Salesforce）』に、Databricks上のPythonコードからデータを送信したい時はどうするの？出口がないからエラーにならない？」

そのために、VNet Injectionと併用して **「NAT ゲートウェイ（Network Address Translation）」** または **「Azure Firewall」** を、パブリックサブネット側に取り付けます。
*   Sparkからの外部通信は、すべて **NATゲートウェイの「たった1つの固定された共通パブリックIP」** を経由して外に出ていきます。
*   これによって、Salesforce等の接続先に対して「うちの会社のこのIPリスト（NATの一つのIP）からしかアクセスさせないでください」という固定IPのファイアウォール制限（IPホワイトリスト化）をかけることが可能になります。
