# Part 4 Week 1②：何でもAPI化する「Model Serving」

前章で紹介した「Foundation Model API」は、あくまで「あらかじめ用意された世界で有名なLLM」を呼び出す機能でした。
しかし、自社で構築した「オリジナルの予測モデル（PyTorch/Scikit-learn）」や「LangChainで組んだ数行のRAGシステムのコード」を、自社のWebサイトからWeb APIとして呼び出せるようにしたい（デプロイしたい）場合はどうするのでしょうか？

それを一瞬で叶えるのが、Databricks最強のデプロイ基盤 **`Model Serving (モデルサービング)`** エンドポイントです。

## 1. 昔の辛い「推論API」構築
数年前までは、データサイエンティストがモデルを作った後、AIエンジニアが以下のような地獄の作業を行っていました。

1. Pythonの `Flask` や `FastAPI` でWebサーバーのコードを1からゴリゴリ書く。
2. そのコードとモデルを「Dockerコンテナ」でパックマンのように包む（Dockerfileを書く）。
3. AWSのECS等にアップロードし、ロードバランサー（通信の割り振り機）を設定する。
4. アクセス集中（スパイク）に備えて、サーバー台数を自動増減させる「オートスケール」のインフラ設計をする。

**➡️ 全部不要になりました。**

## 2. Serverless Model Serving の魔法

あなたがMLflow（Week 3で詳しく解説します）に一度保存したモデルやPythonの予測関数なら、Databricks UIの「Serving」画面から**「Create Serving Endpoint」という青いボタンを1回ポチッと押すだけ**で、前述の①〜④までが数十秒で全自動構築されます。

*   出来上がるもの： `https://adb-xxxx.azuredatabricks.net/serving-endpoints/my-cool-model/invocations` という、**世界中どこからでも即座にPOSTして回答をもらえるあなた専用のURL（REST API）**。
*   このURLを、社内のフロントエンド（ReactのWebアプリ等）のエンジニアに「これにJSON投げたらAIの答えが返ってくるから、あとはよろしく！」と丸投げして連携完了です。

## 3. Scale-to-Zero（スケール・トゥ・ゼロ）の威力

> [!IMPORTANT]
> **💰 インフラ設計の観点：コストの最適化**
> Model Servingは「Serverless（サーバーレス）」という仕組みで動いています。
> 
> もし、作成したAPIに「夜間や休日は誰もアクセスしてこない」という場合、DatabricksのServing Endpointは**「裏側のコンテナ台数（サーバー）を自動的に『0台（ゼロ）』まで減らして、課金を完全に停止する」**ことが可能です。
> 
> （※次にアクセスが来た瞬間に、数秒かけて1台目のサーバーが自動で起ち上がります＝コールドスタート）。
> この設定（Scale-to-Zero）を有効にしておくことで、本番環境の推論APIにかかるコストを、EC2などの常時起動サーバーに比べて劇的に安く抑えることができます。

## 4. Unity Catalog との連携（最強の盾）

Model Servingで公開したエンドポイントは、完全に「Unity Catalog」の支配下に置かれます。
「このモデルAPIは機密度が高いから、営業部門のAさんにだけ `CAN QUERY`（APIを叩く権限）を与え、他部署のBさんからはアクセスさせない」といったセキュリティが、UIから一瞬で設定可能です。
これで、勝手に知らない部署のシステムからAPIを叩かれて課金が爆発するのを防ぐことができます。
