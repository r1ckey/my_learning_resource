# Part 4 Week 3①：LLMのための「泥臭い」運用基盤（LLMOps / MLflow Tracing）

プロンプトを書いて「動いたぞ！」と喜ぶのは最初の一週間だけです。
AIシステムの本格運用が始まると、**「なぜAIはあの時、ユーザーに変な回答をしたのか？どんな経緯であんな嘘をついたのか？」** という証拠・ログ（Traceability）の追跡という泥臭い保守業務が始まります。これが LLMOps の第一歩です。

## 1. RAGのブラックボックス化の恐怖
ユーザーの質問に対し、RAGの裏側では以下のような巨大なパイプライン（チェーン）が走っています。
1.  Pythonがユーザーの文字列を受け取る
2.  PythonがEmbeddingモデルを呼び出してベクトル化する
3.  Vector DBにアクセスして上位3件の社内規程を見つける
4.  その規程を元にPythonが巨大な「プロンプト」を生成する
5.  プロンプトをLlama 3等のLLM APIに送信する
6.  返ってきた回答をPythonが整形する

これらが0.5秒の間に数十〜数百の複雑な分岐を経て実行されるため、後から「どのステップで間違えたのか？」を突き止めるのは（通常の `print()` ログ等では）不可能です。

## 2. MLflow Tracing (オートトレイシング)の衝撃

Databricks社が開発の主導権を握る機械学習トラッキングツール「MLflow」。
最近追加された最強の切り札が **MLflow Tracing (Autologging)** 機能です。

たった1行のコードをノートブックの最初に書くだけで、以降のLangChain等のすべてのAIチェーンが「完璧に可視化（トレース）」されて記録・保存されます。

### 実装例
```python
import mlflow

# AI用オートロギング（トレーシング）機能をONにする
mlflow.autolog()

# ここから下はLangChain等のコードを書くだけ
rag_chain.invoke("出張費はどうやって精算するの？")
```

### 驚異のUI (Trace View)
このコードを実行した後、Databricksの「Experiments（実験）」画面を開くと、**数千文字に及ぶAIの思考プロセスが、完璧なグラフ（木構造）になって保存されています**。

*   **1層目（Rtierever）:** 「出張費・精算」というキーワードに反応して、Vector DBから『第5条 交通費申請のフォーマット.pdf』の内容を引っ張ってきていた。
*   **2層目（Prompt Template）:** 「あなたは優秀な社内AIです。以下のContextをもとに答えて…（全文）」という、LLMに送信された**実際の超巨大プロンプト文字列が、すべての改行・句読点に至るまで一言一句漏らさず保存**されている。
*   **3層目（LLM Call）:** Llama 3が何という回答を返し、それに何秒（レイテンシ）かかり、何トークン消費して、APIに何円課金されたのかが表示される。

> [!TIP]
> 「RAGが変なことを言った！」とクレームが入ったら、MLflowのTrace画面を開いてください。
> ほとんどの場合、「見つけてきたPDFが間違っていた（ベクタ検索の精度不足）」か、「プロンプトへの渡し方が悪くてAIが勘違いした」かのどちらかが、数秒で特定できます。
