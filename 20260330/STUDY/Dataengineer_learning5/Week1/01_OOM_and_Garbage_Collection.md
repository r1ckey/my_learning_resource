# Deep Dive Week 1①：メモリ崩壊（OOM）と時を止める「Garbage Collection」

ビッグデータにおける死亡原因の90%以上は、例外なく **Out Of Memory (OOM：メモリ不足クラッシュ)** です。
Sparkは「データをハードディスクに書かず、全てを高速なメモリ（RAM）上で行う（インメモリ処理）」ため、メモリの使い道（限界点）を知らなければ絶対に使いこなせません。

## 1. Executor（計算する小人）のメモリ領域分割

Sparkのワーカーノード（計算するPC群）にあるメモリ（例：16GB）は、主に以下の2つの巨大な陣地に分割されています。

*   **Storage Memory (保存スペース):**
    あなたが `df.cache()` と書いた時に、頻繁に使うデータをずっと保存しておくための棚。
*   **Execution Memory (作業スペース):**
    JOINやGROUP BY（シャッフル）を行う時に、データを一時的に広げて計算するための机。

> [!WARNING]
> **🚨 初心者のOOM (キャッシュのしすぎ)**
> 初心者は「キャッシュ（保存）すればするほど速くなる！」と勘違いし、巨大なテーブルを全部 `cache()` してしまいます。
> そうすると**「Storage欄がパンパンになり、計算するための『机（Execution）』が1ミリも無くなる」**ため、いざJOINの計算を始めた瞬間に「机にデータが乗り切らない！」とパニックになりOOMで死亡（クラッシュ）します。

## 2. Javaの呪い「Garbage Collection (GC)」

Sparkは「JVM（Java仮想マシン）」という言語基盤の上で動いています。
文字列（JSON等）を大量に処理しようとすると、JVMの中で「10億個のゴミ（一瞬だけ使ってすぐ捨てる変数など）」が発生します。

そしてJVMは時折、**「ちょっとゴミが溜まりすぎたから、一度時間をパキッと止めて、部屋（メモリ）のゴミ掃除をさせてくれ！」**と言い出します。これを **Garbage Collection (GC: ガベージ・コレクション)** と呼びます。

### ⏱️ GC Pause (時が止まる恐怖)
GCの恐ろしいところは、**「ゴミ掃除をしている間は、Sparkの計算処理が1ミリも進まず【完全停止】する」**という点です。
*   「5分で終わるはずの計算が、なぜか20分かかっている。Spark UIの画面を見たら、**15分間はずっとゴミ掃除（GC Time）をしていた**」という、とんでもない非効率が現場で日常的に発生します。

### 🔧 アーキテクトの回避術 (UDFの禁止)
このゴミ（Javaオブジェクト）の大量生産を避けるための絶対原則が、**「Pythonで書いた自作関数（UDF: User Defined Function）を絶対に使うな」**というものです。

Pythonの関数を通すたびに、Sparkは「内部のJava形式のデータ」を「Python形式のデータ」に翻訳（シリアライズ）しなければならず、この翻訳過程で異常な量の「ゴミオブジェクト（メモリの無駄）」が生まれ、即座にOOMとGC停止地獄に陥ります。
