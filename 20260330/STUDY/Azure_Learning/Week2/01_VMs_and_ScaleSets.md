# Azure Week 2①：仮想マシン (VM) と可用性の極限 (Scale Sets)

AZ-104の「Compute（コンピューティング）」領域において、基本となるIaaS（Infrastructure as a Service）である仮想マシン（VM）の**「絶対に落ちない（停止しない）アーキテクチャ設計」**が問われます。

## 1. SLA（サービス品質保証）と可用性の3段階
「オンプレミス（自社サーバー）と違い、クラウドは勝手に壊れて勝手に再起動する」という前提（Design for Failure）でシステムを組む必要があります。
以下の3つの可用性（Availability）の違いを完璧に暗記してください。

### ① 単一のVM（Premium SSD使用）
*   **SLA:** 99.9%
*   一台ポツンと立てただけのサーバーです。Microsoftが裏のハードウェアのメンテナンス等を行う際に、再起動で一時的にオフラインになる可能性があります。

### ② Availability Sets (可用性セット / アベイラビリティセット)
*   **SLA:** 99.95%
*   「2台以上」のVMを束ねる設定です。
*   **仕組み (Update Domain / Fault Domain):** 同じデータセンター（同じ建物）の中で、**「電源やネットワークのラック（棚）を別々にする」**ことで、どれか1つのラックの電源がショートして燃えても、別のラックにあるVMが生き残るように配置を分散する仕組みです。

### ③ Availability Zones (可用性ゾーン / アベイラビリティゾーン)
*   **SLA:** 99.99% (年間のダウンタイムは数分レベル)
*   これが現代の最強アーキテクチャです。
*   **仕組み:** 同じリージョン（例えば「東日本」）の中で、**「数キロメートル〜数十キロメートル離れた全く別の建物・データセンター（Zone 1, Zone 2, Zone 3）」**にVMを分散して配置します。
*   建物の1つが火災や落雷・大洪水で丸ごと破壊された（停電した）としても、別の建物にあるVMが生き残ってサービスを継続します。

> [!WARNING]
> **🚨 Exam Trap (可用性オプションの選択)**
> **[問題]**: 「ある企業がAzure上にWebサーバーを3台構築したい。**『データセンター全体』に及ぶ大規模な障害（停電や火災など）**が発生しても、システム全体が停止しない最も高いSLAを要求している。どの構成にすべきか？」
> 
> *   ❌ Availability Sets を使う (データセンター単位の火災だと一緒に燃えて死ぬので不正解)
> *   ⭕️ **Availability Zones** を使い、3台のVMをそれぞれZone 1, 2, 3に配置する！

## 2. Virtual Machine Scale Sets (VMSS / スケールセット)
突然のバズやTV放送で「100人だったアクセスが、1時間だけ10万人に増えた！」という時に、手動でVMを構築していては間に合いません。

**VMSS（スケールセット）** は、「同じ金型（イメージ）」から全自動でVMをコピーして乱造（スケールアウト）し、暇になったら自動で削除（スケールイン）して課金を止める機能です。

### 最強の機能：Auto-scale (オートスケール判定基準)
VMSSは、何を監視して台数を増やすのでしょうか？

*   **メトリクスベース (CPUやメモリの逼迫度):** 「VM群の平均CPU使用率が75%を5分間超えたら、新しくVMを2台追加（スケールアウト）する」「平均CPU使用率が25%を下回れば、1台減らす（スケールイン）」。
*   **スケジュールベース:** 「金曜の夜21時〜土日の間だけ、強制的に10台に増やす」。

> [!IMPORTANT]
> **🚀 Exam Focus (スケールへの勘違い)**
> 「単体で構築したVM（可用性セットの中に入れているVM）」に対して、「Auto-scaleルールの設定画面（CPUが80%でVMを増やす設定）」を適用することは可能か？
> 
> **[絶対の正解]**: **不可能です。**
> 自動スケールアウト（台数の増減）機能は、最初から「Virtual Machine Scale Sets (VMSS)」として作成されたリソースグループでないと使用できません。「あとから普通のVMをオートスケール対応にする」ことはできないので、要件定義の初日に決断する必要があります。
