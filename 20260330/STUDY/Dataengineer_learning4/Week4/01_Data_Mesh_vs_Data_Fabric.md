# Core Week 4①：真の組織論「Data Mesh」vs 統合の「Data Fabric」

ここまでの旅で、あなたは一人の人間として、完璧なデータプラットフォーム（巨大な一元化されたデータレイクハウス）を作り上げることができました。

しかし、会社がグローバル展開し、社員が数万人になった時。
**「中央のITデータチーム（あなた）」が、全社（営業、マーケ、人事、工場…）から押し寄せる『ダッシュボードを作ってくれ！』『新しいデータソースを追加してくれ！』という数千件の要求の前に立ちはだかる【巨大なシステム上のボトルネック】として崩壊します。**

この「巨大な中央集権の崩壊」を救うための、データアーキテクチャの最終行き着く未来論争（2大バズワード）が、**Data Mesh（データメッシュ）** と **Data Fabric（データファブリック）** です。

## 1. 人間と組織の分散処理「Data Mesh (データメッシュ)」

Zhamak Dehghani氏が提唱した、**「中央集権（モノリス）をぶっ壊し、ドメイン（各部署）に責任を分散させろ」**という、強烈な【組織・社会技術的な思想】です。

*   **課題の根底:** 中央のエンジニア（あなた）は「マーケティングの専門知識（何のKPIが重要か）」も「工場の専門知識（どのIoTセンサーが壊れやすいか）」も知りません。知らないのに、彼らのためにテーブル（スタースキーマ）を設計させられているから、使い物にならず、修正依頼のループで死ぬのです。
*   **Data Mesh の4つの原則（解決策）:**
    1.  **ドメイン主導のオーナーシップ:** データを「中央」に集めるのをやめる。『マーケティング部』は自分たちでデータエンジニアを雇い、自分たちの責任で『マーケ用の最高のデータ』を加工し、保持しろ！
    2.  **データ・アズ・ア・プロダクト (Data as a Product):** ただし、それぞれの部署は、他の部署が「簡単に使える（クオリティが保証され、カタログ化された）」状態の【お弁当箱（プロダクト）】の形で、データを社内に公開（提供）する義務を負う。
    3.  **セルフサービス型のインフラ基盤:** 中央のITチーム（あなた）は、もう彼らの生のテーブルの中身には触らない。「Databricksのクラスター構築」や「dbtのCI/CD環境」といった『自動化されたプラットフォーム環境』だけを整備し、各部署に「はい、この環境を使って好きに作ってください」と提供する基盤チームに徹する。
    4.  **フェデレーテッド（連邦型）ガバナンス:** 各部署が勝手に公開している「お弁当（プロダクト）」をつまみ食いする時に、他部署のデータでもセキュリティ（GDPR等）が担保されるよう、Unity Catalogのような強力な統合監視層（カタログ）で全体を監視する。

➡️ つまりData Meshとは、**「マイクロサービス・アーキテクチャ（ソフトウェア開発の分散化）」を、データの世界（組織論）にそのまま持ち込んだもの**です。

## 2. AIによる自動縫合「Data Fabric (データファブリック)」

一方、Gartnerなどが提唱する Data Fabric は、分散するのではなく**「技術（テクノロジーとAI）の力で、バラバラになったデータシステムを一つの魔法の布（Fabric：布目）のように繋ぎ合わせる」**という【中央集権テクノロジー的な思想】です。

*   **課題の根底:** Data Meshのように「各部署にデータエンジニアを雇え！」と言っても、優秀なデータエンジニアは高給取りで採用不可能であり、現実は各部署でクソみたいなExcelが量産されサイロ化（孤立）するだけです。
*   **Data Fabric の思想（解決策）:**
    「データがAWSのS3にあろうが、Salesforceの中にあろうが、各部署のオンプレミスサーバーにあろうが、**動かさなくていい（抽出・ETLしなくていい）。**
    その代わり、**メタデータ（カタログ情報）をAI（機械学習・GenAI）に24時間監視させ、AIが自動的に『あ、ここのシステムの顧客IDと、あのシステムのCustomer Nameは同じだな』と関連性（ナレッジグラフ）を勝手に生成・推定し、巨大な仮想のデータベース（Logical DWH）として人間に提供する。**」

人間がETLパイプラインを書くのではなく、AIがデータ同士を自動で縫い合わせて（ネットワーク化して）提供する、テクノロジー志向の未来図です。
