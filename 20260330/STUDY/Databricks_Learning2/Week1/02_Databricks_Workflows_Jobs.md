# 認定対策 Week 1②：Databricks Workflows（Jobs）の依存関係

Data Engineer Associate試験では、**Databricks Workflows（タスクの自動管理・オーケストレーション）** のUIの動きや設定値が、「これでもか」というほどしつこく問われます。
特に「Task（タスク）同士の依存関係」と「トリガーの条件設定」は完全な暗記が必要です。

## 1. ワークフローにおける「Task」の直列と並列
Databricks Workflowsの中では、複数の「Task（ノートブックやSQLスクリプト）」を数珠つなぎに（DAGとして）設定できます。

*   **Task A** (データ抽出) ➡️ **Task B** (データ加工) ➡️ **Task C** (データ集計)
このように、前のタスクの完了を待って次のタスクを動かす設定（Depends On）の挙動が試験に出ます。

## 2. 「Run If」条件（前のタスクがどうなったら次を動かすか）
Task B の設定画面には、**「Run If（条件）」** というドロップダウンリストがあり、Task Aの結果を受けてどう動くかを定義できます。試験に出る3大条件を暗記してください。

1. **`All Succeeded` (すべて成功) ★デフォルト設定**
   *   挙動: 依存している**Task Aが「成功(Success)」した場合のみ**、Task Bが起動します。Task Aがエラーで落ちた場合、Task Bは「Skipped（スキップ）」されます。
2. **`At Least One Failed` (少なくとも1つ失敗)**
   *   挙動: Task Aが**「エラー(Failed)」で落ちた場合にのみ**起動する設定です。
   *   使い所: エラーを検知して「Slackに通知を送る」「一時テーブルのゴミを消去する」といった例外処理・クリーンナップタスク(Cleanup Task)専用の設定として試験でよく問われます。
3. **`All Done` (成否にかかわらず完了)**
   *   挙動: Task Aが成功しようが失敗しようが、とにかくTask Aの処理時間が終わったら絶対にTask Bを走らせる設定です。「リソースのシャットダウン」などのタスクに使われます。

> [!WARNING]
> **🚨 Exam Trap (超頻出の引っかけ問題)**
> **[状況]:** Task Aのあとに、Task B と Task C が並列で繋がっている（A -> B, A -> C）。
> Task BのRun Ifは `All Succeeded`。
> Task CのRun Ifは `All Done`。
> **[問題設定]:** ここで、Task A が途中でエラーになって失敗（Failed）しました。さて、ワークフロー全体の挙動はどうなるでしょうか？
> 
> **[絶対の正解]:** Task Bは（Aが成功していないので）**スキップ**される。しかし Task Cは（Aが失敗しても『Done』には変わりないので）**実行される**。
> 「Aがコケたら全てが止まる」と勘違いさせる引っ掛け問題です！

## 3. Workflows間でデータを渡す手法 (Task Values)
「Task Aで得た変数の値（例：今日処理したレコード件数）を、次のTask Bで使いたい」という場合、どうするべきか問われます。

*   ❌ [誤り] `spark.sql("SET変数")` で保持する。 (Taskごとにサーバーが異なるため消えます)
*   ❌ [誤り] S3やDBFSのテキストファイルに一旦書き出して、Task Bで読み直す。 (できなくはないが、ベストプラクティスではない)
*   ⭕️ **[絶対の正解] `dbutils.jobs.taskValues.set()` と `get()` を使う。**
   *   Databricks Workflowsには、タスク間で小さな変数をやり取りするための「Task Values (タスクバリュー)」という専用のAPI（dbutils）があります。試験でこの単語が出たらこれが正解です。
