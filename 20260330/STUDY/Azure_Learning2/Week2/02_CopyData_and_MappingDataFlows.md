# Azure Week 2②：ETLの分離「Copy Activity vs Data Flows」

ADFの中でデータを操作するための様々な箱（アクティビティ）。その中でも、データエンジニアが実務で一番多用する「最強の運び屋」と「最強の加工屋」の2つを知らなければ、無駄な課金と遅延に悩まされることになります。

## 1. 最速の運び屋「Copy Data Activity」

「オンプレミスのMySQLから、1億件のテーブル情報をそのままの形でAzure Data Lake (ADLS Gen2) に持ってくる（CSV等で保存する）」という **（E：抽出）** と **（L：ロード）** のステップ。
これにおいて、ADFの **Copy Data** アクティビティの右に出るものは世界に一つもありません。

*   **完全な並列処理**: 「テーブルを10の塊に分割して、10台のサーバーで並列で一気に引っこ抜く」といったチューニング（並列度 DIU の設定）が可能です。
*   **デメリット**: 「運ぶ専門」なので、**中身の加工は一切できません。**（「年齢が20歳未満を消す」「他のテーブルとJoinする」等は不可）。

## 2. 視覚的な加工屋「Mapping Data Flows」

運び屋（Copy Data）によってADLSの「ブロンズ層（生データ）」に運ばれた汚いデータを、綺麗に整形（T：変換 / トランスフォーム）してシルバー層・ゴールド層に持っていくための箱が **Mapping Data Flow** アクティビティです。

### 🪄 コード(Python/SQL)を書かずにETLが組める
画面上のGUIキャンバスで、アイコンをポチポチつなぐだけで高度なデータ加工が完成します。
*   `Filter` (特定の条件で行を弾く)
*   `Join` (2つのデータを結合する)
*   `Aggregate` (売上を集計する)
*   `Pivot` (行と列を入れ替える)

> [!CAUTION]
> **🚨 Exam Trap (裏側で何が動いているのか？)**
> GUIでアイコンを並べているだけなのに、なぜ数億件のデータが一瞬で処理できるのでしょうか？
> 
> **[DP-203の核心]**:
> Mapping Data Flow の裏側（バックエンド）では、**私たちが配置したアイコンの設計図を自動的に「PySparkスクリプト」に翻訳し、Azure側が用意した管理下の「Databricks (Spark) クラスター」をその場（数分）でポコッと立ち上げ、そのクラスター上で分散処理としてぶん回している** のです。
> （※処理が終わると、Sparkクラスターは静かに消滅・課金停止します。これを『TTL (Time To Live)』によるウォームアップ時間の短縮設定などで最適化します。）

## 3. 🚀 アーキテクトの究極の決断：ADF vs Databricks Notebook

> 「加工（T）は、ADFの『Mapping Data Flow』でポチポチ作るべきか？ それとも、ADFから『Databricks Notebook』アクティビティを呼び出して、Python(PySpark)でゴリゴリ書くべきか？」

これはシニアデータエンジニアにとって永遠のテーマであり、設計思想の見せ所です。

*   **ADF Data Flow の利点:**
    *   Pythonを知らない人（BI担当者やビジネス側）でもデータ加工パイプラインを作れる、メンテナンスできる（ローコード/ノーコード）。
*   **Databricks Notebook (Python) の利点:**
    *   **Git連携とバージョン管理（CI/CD）が美しく組める。**
    *   機械学習モデルの組み込み（MLflow連係）など、複雑すぎるロジック（Part 3, 4で学んだRAGなど）も数行で書ける。
    *   クラスターのSpotインスタンス等を細かく設定・維持できるため、長期的にはパフォーマンス・コスト面で圧倒的に有利。

現代のモダンデータスタックにおける到達点（Part 3, 4 レベルのエンジニア）は、**「データのただの移動（コピー）はADFに任せ、重い頭脳戦（変換加工・T）はすべてDatabricksのNotebookに委譲して呼び出す（AzureとDatabricksの融合アーキテクチャ）」** というのが一つの大正解です。
