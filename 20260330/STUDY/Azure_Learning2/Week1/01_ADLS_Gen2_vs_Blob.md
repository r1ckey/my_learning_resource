# Azure Week 1①：ビッグデータの命綱「ADLS Gen2 と HNS」

データプラットフォームの根幹となる「データレイク」。Azure上でそれを構築するときに、全世界のデータエンジニアが絶対に間違えてはならない「究極のチェックボックス」の存在について学びます。

## 1. Blob Storage と ADLS Gen2 の正体
Azureには「Azure Blob Storage」という安価で大容量のオブジェクトストレージ（AWSにおけるS3のようなもの）が存在します。
そして実は、**「Azure Data Lake Storage (ADLS) Gen2」という独立したサービスは存在しません**。

*   **正体**: ADLS Gen2 とは、ただのBlob Storageに「**階層型名前空間 (Hierarchical Name Space: HNS)**」という特別な機能（魔法）をONにして作成したものの別名です。

> [!CAUTION]
> **🚨 Exam Trap (後から変更できない恐怖)**
> ストレージアカウントを作成する画面の「詳細」タブの中に、ひっそりと `[ ] 階層型名前空間を有効にする` という薄いチェックボックスがあります。
> **これをチェックせずに作成ボタンを押してしまうと、永遠にADLS Gen2にはならず、後からONにすることも不可能です（ストレージを作り直すしかありません）。** 試験でも実務でも、データレイク設計の「1手目」の最重要ポイントです。

## 2. なぜ「HNS (階層型名前空間)」が不可欠なのか？
DatabricksやSynapseでビッグデータを処理する（Sparkを走らせる）時、通常のBlob Storage（HNSがOFF）を使うと、**ファイルの移動や名前変更（Rename）の処理が10倍以上遅くなります**。

*   **HNSが【OFF】の場合（フラットなBlob）**:
    本当はパソコンのような「フォルダ（ディレクトリ）」の概念が存在しません。`/2026/03/sales.csv` という「スラッシュを含んだ長いファイル名」として1列に保存し、画面上だけで『まるでフォルダがあるかのように幻覚を見せている（仮想ディレクトリ）』だけです。
    **➡️ 致命的な欠陥**: `2026` というフォルダの名前を `2027` にRenameしようとすると、中に入っている数千万個のファイル「すべて」を1個ずつコピーして `2027/03/...` として作り直し、元のファイルを全部削除するという、地獄の重い処理（数十分〜数時間）がバックグラウンドで走ります。
*   **HNSが【ON】の場合（ADLS Gen2）**:
    LinuxやWindowsのような「本物のフォルダ（ディレクトリ）」という概念をストレージ内に構築します。
    **➡️ 最強のメリット**: フォルダの名前をパッと書き換えるだけで、中身の数千万ファイルには一切触れず「0.1秒」でディレクトリのRenameや移動（メタデータ操作のみ）が完了します。Sparkは頻繁に一時ディレクトリへの移動（commit）を行うため、これがONじゃないとクラッシュします。
