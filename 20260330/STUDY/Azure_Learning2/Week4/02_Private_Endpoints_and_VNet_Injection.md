# Azure Week 4②：究極の閉域網「Private Endpoint」と「VNet Injection」

Azureインフラ（AZ-104）とデータプラットフォーム（DP-203）を統合する、試験のラスボス技術であり、エンタープライズの現場で最も苦戦する『完全な閉域化・孤立（アイソレーション）アーキテクチャ』の設計です。

「クレジットカード情報や顧客の住所を保存しているデータベース（SQL）やストレージ（ADLS）を、絶対に、1ミリもインターネット（図の枠外）に露出させてはならない」という要件をどう実現するか。

## 1. PaaSというサービスの「弱点」
ADLS、SQL Database、Synapse 等々の「PaaS (Platform as a Service)」は、基本的に「パブリックなインターネット上に存在」します。

（※たとえNSGファイヤウォールで「弊社の会社のIPアドレスからしかアクセスさせない」と制限していたとしても、データ自体はインターネット回線の上（パブリック空間）を一度通過して届けられます）。
これは銀行などの超・厳格な金融システムでは許されません。

## 2. 解決策：Private Endpoint (プライベートエンドポイント)

PaaSを「あなたが作った城（VNet / 仮想ネットワーク）の中へ、無理やり引きずり込む」魔法の機能です。

*   **仕組み:** あなたのVNet（例: `10.0.1.0/24` のSubnet）の中に、ADLS用に `10.0.1.5` という**完全なプライベートIP（LAN内のローカルIP）を持った「分身のネットワークカード（NIC）」を生成して突き刺します**。
*   **結果:** VNet内のVM（サーバー）から `10.0.1.5` にアクセスすると、インターネットに一切出ることなく、Azureの裏の専用線（Microsoftバックボーン）だけを通って、安全にクラウドストレージの中身に直結できます。
*   **最強の盾**: これを設定したADLS等は、設定画面から「パブリックネットワークからのアクセスを完全に【無効(Deny)】」にチェックを入れて封鎖することができます。これで、世界のすべてから遮断された絶対に侵入できない最強の小部屋が完成します。

> [!WARNING]
> **🚨 Exam Trap (DNSという地獄)**
> 私たちのアプリのコードは、ADLSにアクセスする際に `https://my-storage.blob.core.windows.net` というURLを叩きます。
> しかし、Private Endpointを作るとIPアドレスが「裏のLAN用(`10.0.1.5`)」にすり替わっています。
> 
> 何をしないとアクセスできなくなってしまうか？
> 
> **[絶対のトラブル解決ルート]**:
> 「Azure Private DNS Zone」という設定を連動させ、**『このURLを叩いた時は、世界のインターネットのIPではなく、社内の「10.0.1.5」に変換しなさい』というマッピング（名前解決）を手動・自動でオーバーライド（上書き）してあげなければなりません。** 
> 「Private Endpointを作ったのに繋がらない」という問題が出たら、100%「Private DNS Zone の設定が漏れているから」が正解です。

## 3. VNet Injection (Databricksの閉域網展開)

PaaSを既存のVNetに引きずり込む「Private Endpoint」の逆で、**「PaaSのサーバー本体（Databricksの巨大なSparkクラスター群）を、丸ごと私のVNetの中で生成・起動してくれ！」**と要求する機能が **VNet Injection（VNet挿入 / 統合）** です。

*   **使い所**: 「Databricksが計算で使うクラスターは、うちの会社が決めた監視ツールを通った『社内のIPアドレス』から通信してほしいし、NSGでガチガチに出口を縛り付けたい」という大企業の要請。
*   **制約**: 自分のVNet内に、Databricksが自由に使う「パブリック(外への出口用) サブネット」と「プライベート(内側の計算用) サブネット」の **最低2つの空っぽのサブネット** を事前に用意して差し出さなければなりません。
