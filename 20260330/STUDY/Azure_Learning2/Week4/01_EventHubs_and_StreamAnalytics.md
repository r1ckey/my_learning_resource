# Azure Week 4①：激流を捌く「Event Hubs」と「Stream Analytics」

Databricksコースの「Part 3 Week 2: 無限データの防波堤 (Watermarking)」で学んだストリーミングの概念を、Azureの完全純正サービスだけで組み上げるアーキテクチャ設計です。

IoTデバイスやスマホアプリから毎秒1万件飛んでくるログ・センサーデータを「取りこぼさずに受け止める門番」と「それをリアルタイムでさばく処理機」の2つを知らなければ、DP-203は絶対に合格できません。

## 1. 巨大な門番「Azure Event Hubs」
ストリーミングデータにおいて、1秒間に万単位で飛んでくるデータを直接データベース（SQL Server等）に書き込もうとすると、1瞬でDBがクラッシュして火を吹きます。

これを防ぐための「巨大なバッファ（クッション・一時保管庫）」が **Event Hubs** です。
（※オープンソースで言うと Apache Kafka に該当する超巨大サービスです）。

### パーティション（分割）の設計
Event Hubsの中にデータが入ってくると、用意した「パーティション（例えば10個）」のどれかに振り分けられて蓄積されていきます。

> [!CAUTION]
> **🚨 Exam Trap (後出しジャンケンの禁止)**
> **[問題]**: 「Event Hubsを作成する際、パーティション数を『4』に設定しました。しかし、1ヶ月後にアクセスが急増（バズった）したため、パーティション数を『16』に設定変更（スケールアウト）しようとしました。可能ですか？」
> 
> *   ❌ 可能（Azureだからすぐスケールするでしょ）
> *   ⭕️ **不可能 (Basic / Standard プランの場合)**
> 
> **Event Hubsのパーティション数は、「作成時」に決めたら最後、基本プランでは後から絶対に増減させることができません（変更ダイアログがグレーアウトします）。**
> よって、「将来の最大ピーク（テレビに出た時など）」を見越して、最初から余裕を持ったパーティション数で設計する（あるいは高額なPremium/Dedicatedプランを契約する）必要があります。

## 2. リアルタイム処理機「Stream Analytics」

Event Hubs（門番）が受け止めた「今その瞬間の無限の水流（データ）」に対して、**「SQL風のクエリ」**を使ってリアルタイムに集計を行い、Power BIのダッシュボード等に即座に表示させる機能です。

```sql
-- 5分ごとの窓（Tumbling Window）で、車のスピードセンサーの平均値を出し続けるリアルタイムSQL
SELECT
    CarID,
    AVG(Speed) AS AverageSpeed,
    System.Timestamp() AS WindowEnd_Time
INTO
    [出力先：PowerBIのダッシュボード]
FROM
    [入力元：Event Hubs]
GROUP BY
    CarID,
    -- 🌟 必須の概念: ウィンドウイング関数（5分で区切って切り落とす）
    TumblingWindow(minute, 5)
```

### ウィンドウ（窓）の種類を暗記する
DP-203試験では、どのウィンドウ関数を使うべきかの選択肢が必ず出ます。
*   **Tumbling Window (タンブリング):** 「12:00〜12:05」「12:05〜12:10」と、絶対に被らない（重ならない）固定の切り落とし箱。
*   **Hopping Window (ホッピング):** 箱が被る（例: 5分間の平均を、1分ごとに計算して更新し続ける）。
*   **Sliding Window (スライディング):** イベントが発生した（車が通った）時点から前後の箱を作る。
*   **Session Window (セッション):** 「ユーザーがログインしてマウスをポチポチしている間」= 1つの箱。マウス操作が『5分間途絶えたら』タイムアウトとしてスパッと箱を切り捨てる（動的な窓）。
