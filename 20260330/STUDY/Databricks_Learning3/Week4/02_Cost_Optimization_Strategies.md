# Part 3 Week 4②：クラウド破産を防ぐ「コスト最適化・生存戦略」

Data Engineerの真の評価は、「動くものを作ったか」ではなく**「月額数千万円かかるクラウドのサーバー代（AWS EC2 / Azure VM課金）を、いかにパフォーマンスを落とさず半分に削ったか」**で決まると言っても過言ではありません。
大規模データ基盤のコストを劇的に切り詰める、シニアアーキテクト必須のコスト最適化戦略を伝授します。

## 1. ジョブコンピュート vs 汎用コンピュートの徹底
Part 2で暗記した「Job Compute（ジョブクラスター）は、All-Purpose（汎用クラスター）に比べてDBU単価が半分以下で済む」という知識の実践です。

*   **戦略:** 本番の自動化ワークフロー（深夜バッチなど）において、Interactive/All-Purposeクラスターをアタッチして使い回すことは**金銭的な大罪**です。
*   必ず、Workflow（Jobs）の設定で「**Job Computation（新規に立ち上げて即終了するクラスター）**」を選択してください。これだけで基本料金が半額以下になります。

## 2. 驚異の割引率「Spot Instance（スポット・インスタンス）」の活用

Azure や AWS などのクラウドプロバイダーは、「今誰も使っていなくて余っているサーバーマシンの空き領域」を、通常価格の **70%〜90%オフ（破格の安売り）** で提供しています。これが「Spot Instance」です。

ただし、安売りしている代わりに**「もし他のお金持ちのお客さんが定価でサーバーを使いたいと言ってきたら、あなたのサーバーは数分の猶予で強制終了（Preemption / 回収）されますよ」**という極めて危険なデメリットを持っています。

### どうやって安全に使うのか？
Sparkの「分散処理・フェイルオーバー（障害からの自動復旧）」という最強の性質と、Spot Instanceは恐ろしく相性が良いです。

*   **Driver（司令塔ノード）の設定**: 司令塔が死ぬとJob全体が完全終了してしまうため、Driverノードには絶対にSpotを使わず、通常の定価サーバー（On-Demand）を指定します。
*   **Worker（計算する奴隷ノード）の設定**: このノード群を「100% Spot Instance」で構成します。
*   **結果**: もし計算の途中でクラウド側からWorkerが数台強制終了されたとしても、Driverは全くパニックにならず「ワーカーが2台落ちたか。じゃあ生き残っている残りのノーカーにさっきまでのタスクを再割り当てするね」と自動復旧して計算を続行してくれます！

> [!IMPORTANT]
> **🚀 アーキテクトの判断基準**
> 「絶対に○時○分に終わらないと、証券会社の取引に支障が出る」というような数分の遅延（再計算）も許されない超クリティカルなジョブにはSpotを使ってはいけません。
> しかし、「毎日1回の売上集計で、多少終了時間がブレでも誰も文句を言わないパイプライン」や「テスト環境の処理」には、積極的にSpotワーカーを採用することで、会社のインフラコストを数千万円単位で平気で削減できます。

## 3. Serverless（サーバーレス）コンピュートへの移行
自分たちでスポットだオンデマンドだとマシンの形（EC2の型番）を選定する時代すら終わりつつあります。

現在Databricksが最も推奨しているのが **Serverless SQL Warehouses** や **Serverless DLT** です。
これは裏側にVM（仮想マシン）が常駐せず、アクセスが来た瞬間・クエリが走った瞬間だけDatabricks側の巨大なリソースプールから一瞬で環境が割り当てられ、クエリが終わったら秒速・分速でシャットダウン（スケール・トゥ・ゼロ）し、「アイドル時間（何もしていない待ち時間）の課金を完全にゼロにする」仕組みです。

*   「夜間バッチが3時間で終わった後、朝までクラスターの電源を切り忘れてずっと課金メーターが回っていた…」
という、クラウド破産の最大原因であるヒューマンエラーを、アーキテクチャレベルで撲滅します。
