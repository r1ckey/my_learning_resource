# Part 3 Week 1②：OOMとDisk Spillの回避、そしてCacheの真実

ビッグデータを扱うインフラ設計において、「メモリ管理」は避けて通れない最大のテーマです。データ量が想定を超え始めたときに発生するクラッシュ（OOM）と劇的な遅延（Spill）の原因を突き止め、解決するアプローチを学びます。

## 1. Disk Spill (ディスクへの溢れ) は「死の予兆」
Sparkのワーカーノード（Executor）は、与えられたTaskをRAM（メモリ）上で処理します。
しかし、JOINの際などに「特定のワーカーに与えられたデータ量が、そのワーカーのRAMの限界を超えた」とき、直ちにクラッシュ（OOM）するわけではありません。

Sparkは生き残るために**「入り切らない中間データを一時的にローカルのHDD/SSD（ディスク領域）に逃し、少しずつ入れ替えながら計算を続ける」**という動作を行います。これを **Disk Spill (ディスク・スピル)** と呼びます。

*   **影響:** ディスクへの読み書きはメモリ処理に比べて**1万倍遅い**ため、クエリの実行時間が数十倍に跳ね上がります（数分で終わるものが数週間レベルの速度に低下します）。
*   **アーキテクトの確認:** Databricksの「Spark UI」を開き、赤い警告アイコンとともに `Spill (Disk)` という項目に数値（GB単位など）が出ていたら、そのパイプラインは崩壊寸前です。

### 🛡️ Spillの解決アプローチ
1.  **コンピュートの増強**: 最も簡単な力技です。ワーカーマシンのインスタンスタイプを、よりメモリ容量の大きいもの（例: Mem optimized インスタンス）に変更します。
2.  **パーティション数の増加 (Repartition)**: `df.repartition(400)` のように意図的にタスクの分割数を増やします。100人で運べない巨大な岩でも、400の小石に砕いて400人で運べば、1人が抱えるメモリ量は減り、Spillを防ぐことができます（※前章のSalting技法もこのアプローチの一種です）。

## 2. 実務アンチパターン：思考停止での `df.cache()` 多用
「遅いからとりあえずキャッシュ（メモリへの保存）を使おう」というのは、初心者が最も陥りやすいアンチパターンです。

```python
# 🚫 現場でよく見る「無駄な/有害な」キャッシュ
df = spark.read.table("huge_raw_table")
df_filtered = df.filter("age > 20")
df_filtered.cache() # ←意味がないどころか、他人のメモリを奪っている！
display(df_filtered)
```

### なぜ `cache()` を使ってはいけないのか？
ビッグデータ環境におけるクラスターのメモリは有限の「超高級車」です。
`cache()` を実行すると、そのDataFrameのデータがワーカーのメモリにドンと常駐します。すると、**他の複雑なJOIN処理などで使うべきメモリ領域が圧迫され、前述の『Disk Spill』を意図的に引き起こす最悪の原因になります。**
また、不要になった際に `df.unpersist()` で消去するのを忘れると、永遠に居座り続けます。

> [!IMPORTANT]
> **✅ `cache()` を「絶対に使わなければならない」唯一の条件**
> 
> **「全く同じ重いDataFrame（Transformationの結果）を起点にして、複数の『Action（書き込み、display計算、count等）』を分岐させて実行する場合」のみ**使用します。
> 
> ```python
> # [正解] 時間のかかるベーステーブルを作る
> base_df = heavy_join_and_aggregation().cache()
> 
> # Action 1: 売上が高い顧客を出力する
> top_customers = base_df.filter("sales > 100").write.save(...)
> 
> # Action 2: 「全く同じbase_dfを使って」地域ごとの集計を書き出す
> region_summary = base_df.groupBy("region").count().write.save(...)
> ```
> 
> この場合、`base_df` の計算（例：10分かかる）は、Action1とAction2で**それぞれ実行されて合計20分かかってしまいます**。
> `cache()` を挟んでおけば、Action1で計算した結果がメモリに残り、Action2は「0秒で取得したメモリ上のbase_df」から処理を始められるため、無駄な再計算を完璧に防ぐことができます。
