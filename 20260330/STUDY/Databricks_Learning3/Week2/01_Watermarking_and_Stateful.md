# Part 3 Week 2①：無限データの防波堤「Watermarking」

Auto Loaderなどを利用した「ストリーミング処理」。非常に強力ですが、もしストリーミングデータに対して **`GROUP BY` による集計（例えば、1時間ごとのユーザーログイン回数を数え続ける）** を行おうとすると、システムの根本を揺るがす「状態管理（Stateful Processing）」の壁にぶつかります。

## 1. ストリーミング集計の「State（状態保持）」問題
バッチ処理と違い、ストリーミングは「いつ終わるかわからない無限のデータ」を処理し続けます。
もし「13:00〜14:00の枠」のアクセス数をカウントしていた場合、14:00を過ぎても「もしかしたらネットワーク遅延によって、13:59のデータが15:00に遅れて届くかもしれない」ため、システムはいつまで経っても「13:00の集計バケツ」をメモリ（State Store）の中に維持し続けなければなりません。

*   **結果**: 日数が経過すると、過去のすべての時間枠のバケツがメモリ上に積み上がり、**最終的にクラスターがOut Of Memory (OOM)を起こして完全に沈黙・崩壊します。**

## 2. Watermark（ウォーターマーク/遅延許容限界）の魔法
この破綻をシステム的に防ぎ、メモリを定期的にゴミ箱へ捨てる（クリーンアップする）ための宣言が **Watermark（ウォーターマーク）** です。

Watermarkとは、「**最も新しい到着データから数えて、どれくらい前の遅延までなら許してあげるか**」の限界線の設定です。

```python
import pyspark.sql.functions as F

# ストリームデータの集計（窓関数との組み合わせ）
aggregated_stream = streaming_df \
    .withWatermark("event_time", "2 hours") \
    .groupBy(
        F.window("event_time", "1 hour"), # 1時間ごとの枠(バケツ)で集計
        "user_id"
    ) \
    .count()
```

### どう動くのか？（メモリ保護の仕組み）
上記の `.withWatermark("event_time", "2 hours")` と設定した場合：

1. 今システムが受け取った最も新しいデータの時間が「17:05」だったとします。
2. Sparkは「2時間前である『15:05』をウォーターマーク（最終防衛線）として引きます。
3. この瞬間、**「15:00の集計バケツ（14:00〜15:00枠）はもう絶対に変動しない」と判断し、メモリ上のState Storeからバケツを破棄して最終確定のデータベースに書き込みます。**
4. もし、あとから「14:30のデータです！」という遅延データが到着しても、防衛線よりも古い「期限切れのゴミ」として安全に無視（Drop）されます。

> [!IMPORTANT]
> **🚀 アーキテクトとしての現場の決断**
> 実務で `Spark Streaming` （または `Delta Live Tables` のストリーミング）を用いて**Window集計やストリーム同士のJoin**を行う場合、**`withWatermark` の設定は【必須（マスト）】です。** これを忘れると、本番環境のストリーミングシステムは数週間後〜数ヶ月後に必ず死にます。
> 
> 「ビジネスの要件として、遅延データは何時間後まで集計に反映させるべきか？」という業務ヒアリングから「2 hours」といった値を導き出すのがデータエンジニアの腕の見せ所です。
