# Part 7：リアルタイム・ストリーミングとKafkaの心臓 ロードマップ

数時間遅れても許される「日次バッチ（Batch）」の世界から抜け出し、**「今スマホで『いいね』を押した瞬間に、地球上の全てのサーバーにその事実を同時配信（1秒以内の遅延）する」**という【究極のリアルタイム領域】のメカニズムです。

この世界を支配している絶対的インフラが、LinkedIn社が生み出した分散メッセージキュー **Apache Kafka (アパッチ・カフカ)** です。（※クラウド版では AWS Kinesis や EventHubs に相当します）。

1. **Week 1: Apache Kafka 完全解剖学（TopicとPartition）**
   * Kafkaはデータベースではない。「絶対に消えない強固なメモ帳」であるという哲学。そしてそれを100台のPCで並列処理するための「レーン分割（Partition）」の概念。
2. **Week 2: オフセット（栞）管理と「Exactly-Once（厳密に1回）」の壁**
   * スマホの電波が途切れた時、「クレジットカードの決済処理」が通信エラーでダブって2回実行される（二重決済バグ）のを、分散システムはいかにして物理的に防ぐのか？
3. **Week 3: 時間の相対性理論（WatermarkとState）**
   * 「実際の出来事（イベント時間）」と「サーバーに届いた時間（処理時間）」のズレ。「3時間も遅れて今届いたクリックデータ」を、どうやって『3時間前の集計結果（テーブル）』に割り込ませて計算し直すのかという、ストリーミング最大の鬼門。
4. **Week 4: イベント・ソーシング（究極のマイクロサービス組織論）**
   * 「現在のステータス（結果）」を記録するのをやめろ。「過去に起きたすべての出来事（イベント）」を資産として永遠に保存し続けるという、現代アーキテクチャの最終形態。
