# Core Week 2①：歴史の改ざんを防ぐ「SCD (ゆっくり変化する次元)」

スタースキーマ（前章）を設計した瞬間、必ず経営陣やアナリストから文句が出る**「データの矛盾（バグ）」**が存在します。
それが、「時間の経過とともに変化する顧客の情報」をどう設計の中で管理するかという問題です。

## 1. 炎上事故のケーススタディ
「Aさん」という常連顧客がいます。彼女は2025年まで「東京都」に住んでおり、たくさん買い物をしました。
そして2026年の1月に「大阪府」に引っ越し（プロファイルを変更し）、さらに買い物をしました。

*   **[DBを単純にUPDATE（上書き）してしまった場合]**:
    顧客ディメンションテーブルの住所を「大阪府」で上書き（UPDATE）すると、**アナリストが「2025年（去年）の東京都の売上合計」を集計した時、Aさんが当時東京で買った売上が全て「大阪府の過去の売上」として突然集計されてしまい（歴史の改ざん）、前年度のレポート数字がバグります。**

この「たまに（ゆっくりと）更新される顧客などの属性データ」を、どうやってデータウェアハウス内で扱うか？この世界的標準プロトコルが **SCD (Slowly Changing Dimensions)** です。

## 2. SCD (緩やかに変化する次元) の種類

代表的な3つの種類と、DWHでの対応（面接で必ず聞かれます）を完璧に覚えてください。

### 🔴 Type 1: 上書き（Overwrite）
過去の履歴を完全に無視し、古いデータを消して最新の値（大阪府）で上書き（UPDATE）します。
*   **用途:** 「誤字脱字の修正」や「歴史的な変化を気にしなくてよい項目」にのみ使います。

### 🟢 Type 2: 行の追加（Add new row / 履歴の完全版保持）
**データウェアハウスの「黄金律（ゴールデンスタンダード）」です。**
元の「東京都」のデータ（行）は一切消さず（上書きせず）、新しく「大阪府」のAさんのデータ（行）をインサート追加します。これを管理するため、**システムが自動で発行する「サロゲートキー (代理キー)」**と、「有効開始日・有効終了日（またはフラグ）」の専用カラムを付け足します。

| Surrogate_Key(PK) | 顧客ID | 氏名 | 住所 | Valid_From | Valid_To | Is_Current |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `Key-A1` | 100 | Aさん | 東京都 | 2020-01-01 | **2025-12-31** | False |
| `Key-A2` | 100 | Aさん | 大阪府 | **2026-01-01** | `9999-12-31` | True |

*これにより、2025年の売上ファクトテーブルは昔のAさん(`Key-A1`)とJOINされ、2026年以降の売上ファクトテーブルは新しいAさん(`Key-A2`)とJOINされるため、歴史の改ざんが完璧に防がれます！*

### 🟡 Type 3: 列（カラム）の追加（Add new column）
過去と現在の「最新と1個前の2種類だけ（Before/After）」知りたい場合に、行ではなく「前の住所」「今の住所」という2つの『列』を作ります。しかし拡張性がないためDWHではあまり使われません。

## 3. Data Vault (データヴォルト) 2.0 のパラダイム

数千ものテーブルが存在する「巨大企業（銀行・グローバル企業）」のエンタープライズ領域になると、スタースキーマ（キムボール手法）の設計すら、システム改修のたびにテーブル構造が全壊するという限界を迎えます。
そこで生まれた最新の設計手法が **Data Vault（データヴォルト：データの金庫）** アーキテクチャです。

**「変更されるものと、永遠に変更されないものを、物理的に別のテーブルに切り刻む」**

1.  **Hub (ハブ):** ビジネスのコアとなる「絶対に変わらない背番号（キー）」だけを保存する中心テーブル。（例：顧客IDとハッシュ値のみ）
2.  **Link (リンク):** ハブとハブの「関係性（例：顧客が商品を買った事実）」だけを保存する交差点。
3.  **Satellite (サテライト):** 具体的な情報（住所、名前など『変化（Type 2）しうるもの』）やタイムスタンプを保存する外周。

「属性（名前や住所など）が変わったら、Hub（中心）には一切触れず、SatelliteのテーブルにただひたすらINSERT追加していくだけ」という、アジャイル開発とクラウドデータレイクの思想に完全に一致した（UPDATEを行わない挿入特化型の）強烈なアーキテクチャです。
