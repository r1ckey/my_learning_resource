# Deep Dive Week 1②：Deleteの亡霊を見逃さない「CDC (Change Data Capture)」

MySQLやPostgreSQLといったメインの業務データベース（OLTP）から、分析用のDWHへデータをコピーする時。「差分（新しく増えた分・変わった分）」だけを引っこ抜きたい場合、どう設計しますか？

## 1. 初心者の設計：WHERE updatedAt > 昨日

「簡単ですよ！各テーブルに必ず『最終更新日時（`updated_at`）』の列を作ってもらいます。そして、真夜中にバッチ（Airflow）を回して、**`SELECT * FROM users WHERE updated_at > '昨日の夜中0時'`** とクエリを投げて、引っかかったやつだけを抽出（Extract）すれば差分が取れます！」

これは立派なアプローチ（差分抽出：Incremental Load）です。しかし、この手法には**「システムを破綻させる致命的な欠陥」**がひとつ存在します。

*   **[致命的な弱点：Hard Delete (完全物理削除) の検知不能]**:
    もし、あるユーザーが「退会ボタン（アカウント削除）」を押し、DBのシステムが `DELETE FROM users WHERE id = 100` と実行してしまったとします。
    ➡️ これが行われると、**レコード（行）そのものが消滅してしまうため、夜中のバッチで `SELECT WHERE updated_at` を投げても『削除されたという事実』を抽出することが一切不可能になります（データが取得できない）。**
    結果として、「DWH側の分析サーバーには、退会した幽霊ユーザーのデータがいつまでも残り続ける（バグる）」事態に陥ります。

## 2. データベースの深淵を覗く「CDC (Change Data Capture)」と「Debezium」

この「DELETEが検知できない」「SELECTを投げまくるとメインのDBが重くなる」という2つの大問題を完全に打ち破る、現代における抽出の決定版アーキテクチャが **CDC (Change Data Capture：変更データキャプチャ)** です。

### ⚙️ CDC の仕組み（Transaction Log の盗み見）
すべてのリレーショナル・データベース（MySQLやOracleなど）は、データがクラッシュした時の復旧用に、「SQLでレコードが変更されるよりもさらに奥深く（OSに近い部分）」で、**すべての変更操作（INSERT, UPDATE, DELETEの履歴）を順番にひたすらディスク（ログファイル）に書き出しています。**

* MySQLの場合は **Binlog (Binary Log)**
* PostgreSQLの場合は **WAL (Write-Ahead Logging)** と呼ばれます。

**Debezium (デベジウム)** などの専用CDCツールは、この「データベースの奥底に書かれているログファイルそのもの」を常時（24時間）盗み見・監視します。

1.  **リアルタイム検知**: アプリ側で `DELETE FROM users WHERE id = 100` と叩かれた瞬間、Debeziumはログの書き込みを検知します。
2.  **イベントのストリーム化**: デベジウムは「あ、100番が削徐（Delete）されたぞ！」という『操作イベント（メッセージ）』に変換し、瞬時に **Apache Kafka（メッセージキュー）** に送信します。
3.  **DWH側の処理（消費）**: DWH側のストリーム処理プログラムがそれを受け取り、「なるほど、向こうで100番が消されたのね」と理解し、DWH内のデータを正確に削除（あるいは削除フラグを立てて同期・UPSERT）します。

> [!IMPORTANT]
> **🌟 CDC（Debezium + Kafka）がもたらしたもの**
> いつ来るか分からないバッチの定期ポーリング（SELECT文の連打）から、**「変更があった瞬間に、DB側からプッシュ（送信）されてくる完全リアルタイムな変更ストリーム」**へと抽出のパラダイムが完全に変化しました。
> 負荷も小さく、DELETEされたという履歴も絶対に見逃しません。現代のリアルタイム分析基盤の「入り口（Extract）」は、例外なくこのCDCアーキテクチャで構築されます。
