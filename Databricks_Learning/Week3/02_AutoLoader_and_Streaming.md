# 第3週②：Auto Loaderと無限にスケールする増分処理（ストリーミング）

前章（メダリオン）でアーキテクチャの箱を決めたら、次はその箱と箱の「データの受け渡し（インジェスチョン）」パイプラインを作ります。
ここでは「毎日・毎秒、クラウドストレージにファイルが無限に届き続ける状況」をどうやって効率的に処理するか（特にBronze層への取り込み）に焦点を当てます。

## 1. 旧来のバッチ処理の限界
クラウドストレージの特定ディレクトリ（例: `s3://raw-landing-bucket/logs/`）に、毎日JSONファイルが追加されるとします。
従来の方法（`spark.read.json("s3://.../*")`）で実行すると、バッチジョブが起動するたびに「ディレクトリ内のすべてのファイルリストを取得（LIST操作）」し、「今回のジョブで対象となったファイルと過去のファイルを比較」しなければなりません。
**ファイルが数百万個に達すると、この最初の「ファイルリスト取得（ディスカバリ）」だけで1時間以上かかり、処理が完全に破綻します。**

## 2. Databricks Auto Loader (`cloudFiles`) の仕組み
この課題を根本から解決するのが、Databricksが最も推している「**Auto Loader (オートローダー)**」機能です。

Auto Loaderは、S3などのクラウドストレージに「新しく到着したファイルだけ」を高速で検知し、未処理のインクリメンタル（増分）データのみを取り込むストリーミング機構です。
内部では以下の2つのモードで動きます。

*   **Directory Listing モード (デフォルト)**: 定期的にディレクトリを最速でスキャンし、過去に処理した履歴（Checkpointsディレクトリ内のRocksDB）と照合します。数十万ファイル程度までならこれで高速に動きます。
*   **File Notification モード (エンタープライズ向け)**: S3なら「AWS SQS（キュー）」や「EventBridge」を背後で自動設定します！ ファイルが置かれた瞬間にクラウド側から「置かれたよ」という通知だけを受け取り直接読みに行くため、**フォルダ内に数億個のファイルがあろうと、数秒で新ファイルだけを特定**して読み込みます。

```python
# 実務でのAuto Loader(cloudFiles)の基本コード

# 1. 読み込み (ストリーミングとして定義する)
raw_df = spark.readStream.format("cloudFiles") \
    .option("cloudFiles.format", "json") \
    .option("cloudFiles.schemaLocation", "/tmp/checkpoints/schema/") \ # 新しい列が増えたら自動推論
    .load("/mnt/raw_landing_zone/")

# 2. 書き込み (Bronze層へ)
raw_df.writeStream.format("delta") \
    .option("checkpointLocation", "/tmp/checkpoints/bronze/") \ # 「どこまで読み終わったか」を完璧に記憶する
    .trigger(availableNow=True) \ # ★実務で最強のオプション (後述)
    .toTable("catalog.schema.bronze_logs")
```

## 3. 実務のベストプラクティス： `Trigger Available Now`

「ストリーミング」と聞くと、「クラスター（サーバー）を24時間稼働させっぱなしにして、リアルタイムで待ち受けなければならないのか？ そんなの費用が持たない」と思われがちです。

ここで登場するのが、構造化ストリーミングの最強のオプション **`Trigger(availableNow=True)`** です。

これは「**バッチクラスターでストリーミング機能を使い倒す**」というアプローチです。
1. 深夜1時にバッチジョブ（クラスター起動）。
2. このコードが走ると、Auto Loaderは過去のCheckpointを読み、「前回の実行から現在までに溜まった新ファイル（差分）」を**全て一度だけ**読み込み・処理します。
3. 処理が終わったら、ジョブ終了とともに自らストリーミングを停止し（クラスターもシャットダウン）、次の実行まで眠りにつきます。

> [!IMPORTANT]
> **実務での判断基準**
> 単純な日次バッチ処理（Daily Batch）であっても、通常の `spark.read` を使って「昨日の日付フォルダだけを読む」ように開発するのではなく、**最初からAuto Loader + `Trigger availableNow=True` のストリーミング構文で設計する**のがモダンデータエンジニアリングの定石です。
> これにより、深夜バッチが失敗して数日止まっていたとしても、次回起動時に「未処理分を全て勝手に再開・消化」してくれます。エンジニアが手動で「13日分と、14日分をパラメーターで流し直して…」操作をする必要が完全になくなり、運用工数が激減します。
