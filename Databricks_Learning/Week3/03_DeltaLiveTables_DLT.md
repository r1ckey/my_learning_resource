# 第3週③：Delta Live Tables (DLT) とデクララティブ基盤の真髄

ノートブックにストリーミング（Auto Loader）を使ったETLコードを書けるようになっても、複数の巨大なデータパイプラインを運用していると必ず限界が来ます。
「あるノートブックは毎時動かして、別のノートブックは毎日動かす」「途中でエラーが出たら、次のジョブ（ノートブックタスク）にどう伝えるか？」「テスト環境と本番環境でストレージパスをどう書き直すか？」等、インフラ運用・依存関係の管理の**コード外の泥臭い対応**に時間が割かれます。

これらを解決し、データエンジニアリングの自動化を極限まで押し上げたのが「**Delta Live Tables (DLT)**」です。

## 1. 手順(How)から宣言(What)へのパラダイムシフト
旧来のETLは「**How（どうやって作るか）**」を記述していました。
*   *「AフォルダからJSONを読み込んで、XとYに変換して、一時保存して、それを別ノートブックで読み込み、Bテーブルに書き込むための空箱を作り... もし失敗したらリトライを3回する」*

DLTは「**What（私はどんなデータのテーブルが欲しいか）**」だけを記述する（デクララティブ／宣言型パイプライン）仕組みです。
*   *「私はAというテーブルから不要な空白を取り除き、クリーンになった『Bテーブル』が欲しい。おしまい」*

これだけをコード（SQLやPython）で宣言しておけば、Databricksエンジンが背後で**DAG（有向非巡回グラフ：データの依存関係と流れる順序）を自動で作成し、インフラのプロビジョニング、Auto Loaderによるストリーミング処理、エラー復旧をすべて勝手に行ってくれます。**

## 2. Expectations による強固なデータ品質管理 (DQC)

認定試験 `Data Engineer Associate` において、**必ず出題され合否を分ける**のがDLTの制約（Expectations - エクスペクテーション）機能です。

Silver層で「IDがNULLのレコードは絶対に排除したい」「金額がマイナスだったら何かがおかしいのでジョブ全体を止めたい」というデータ品質ルールを、DLTではテーブル定義に一行付け加えるだけで自動監視（かつダッシュボード化）してくれます。

制約違反時の対応（パニック時のアクション）には**3つのレベル**があり、実務でもこれを使い分けます。

### 🟡 レベル1：監視するだけ (Dropしない・止めない)
```sql
CREATE STREAMING LIVE TABLE silver_users (
  -- CONSTRAINT [制約名] EXPECT ([条件式])
  CONSTRAINT valid_age EXPECT (age > 0)
) AS SELECT * FROM stream(LIVE.bronze_users);
```
*   **挙動:** `age` がマイナスの異常行があっても、そのままSilverテーブル（下流）へ流します（保存します）。ただし、DLTの品質監視画面には「何行の違反があったか」のメトリクス（警告）がカウントされ記録されます。最初はこれで本番データの汚さを調査します。

### 🟠 レベル2：不正な行をポイ捨てして続行 (Drop Row)
```sql
CREATE STREAMING LIVE TABLE silver_sales (
  CONSTRAINT valid_id EXPECT (id IS NOT NULL) ON VIOLATION DROP ROW
) AS SELECT * FROM stream(LIVE.bronze_sales);
```
*   **挙動:** `id` がNULLの行は「ゴミ」とみなして捨て去り、**正しい行に関するデータ処理だけを続行して**Silverテーブルに保存します（クリーンな状態を保つ）。

### 🔴 レベル3：異常検知で即座にパイプラインを止める (Fail Update)
```sql
CREATE STREAMING LIVE TABLE gold_financial_report (
  CONSTRAINT valid_amount EXPECT (total_amount >= 0) ON VIOLATION FAIL UPDATE
) AS SELECT * FROM LIVE.silver_sales;
```
*   **挙動:** 会計・財務システムなど、「たった1行でも不正データがあったら、それを用いて作られたダッシュボードを社長に見せるわけにはいかない」という絶対の品質が求められる場合に設定します。1行でもマイナス金額が入ってきた瞬間、パイプラインの更新全体の実行を**強制ブロックして失敗（エラー停止）**させ、エンジニアにアラートを飛ばさせます。

> [!TIP]
> **実務でのアーキテクチャの棲み分け**
> - 「手動のノートブックETL（Week 2）」＝ 柔軟で複雑なアドホック処理や小規模バッチ向け。
> - 「DLT（Delta Live Tables）」＝ プロダクション（商用本番）として長期稼働させる、数十〜数百のテーブルが絡み合う全社の大規模データパイプライン向け。運用者が退職しても仕組みが自動で回る堅牢さがあります。
