# 第4週①：Unity Catalog (UC) による最強のデータガバナンス

これまでの3週間で、私たちは「高速で」「壊れにくく」「美しい」メダリオンアーキテクチャのデータ基盤を作ってきました。しかし、いざこれを全社（1000人の社員）へ展開しようとした瞬間、最大の壁にぶち当たります。
**「誰に、どのデータを見せていいのか（見せてはいけないのか）」**というガバナンスとセキュリティの問題です。

## 1. レガシーなHiveメタストアの限界（なぜUCが必要か？）
従来のDatabricks（ワークスペースごとのHiveメタストア）では、「テーブルA」という実体が各ワークスペース（A部署用、B部署用など）に完全に閉じ込められていました。
*   **問題点:** B部署が「A部署のテーブルA」を使いたい場合、データを丸ごとコピーしてB部署用のS3（ストレージ）とワークスペースに持ってくる必要があり、二重管理（サイロ化）と莫大なコピーコストが発生していました。

## 2. Unity Catalog のアーキテクチャ（3層構造）
Unity Catalog (UC) は、すべてのワークスペース、すべてのアカウントにまたがる**単一の巨大なガバナンス・レイヤー（カタログ）**です。

UCを有効にすると、データへのアクセスパスはこれまでの「テーブル名だけ」から、**「3層名前空間」**へと昇華します。これはRDB（SQL Server等）に非常に近い概念です。

`SELECT * FROM [カタログ名].[スキーマ名].[テーブル名]`

*   **カタログ (Catalog)**: 最上位のコンテナ。物理的な環境（例：`prod`＝本番環境、`dev`＝開発環境、`sandbox`＝実験場）で分けるのが実務のベストプラクティスです。特定のワークスペース（開発部用）からは `dev` と `sandbox` カタログだけ見え、本番ジョブからは `prod` を参照する、といった強固な分離ができます。
*   **スキーマ (Schema)**: データベースに相当。部署や機能（例：`sales`, `marketing`, `hr`）で分けます。
*   **テーブル (Table)**: 実際のデータ（`daily_summary` 等）。

## 3. 実務での判断：「マネージド」か「外部」か？
UCにおいてデータエンジニアが最初に設計を迫られるのが、「このテーブルの実体（クラウド上のParquetファイル）は、Databricksに完全に管理させるか、それとも自分たちのストレージ上に残すか」です。

### 🛡️ マネージドテーブル (Managed Tables)
*   **定義:** データの実体（S3ディレクトリ）も、メタデータ（列などの情報）も、全てUnity Catalogが作成した管理下ストレージに任せるテーブル。
*   **特徴:** `DROP TABLE` を実行すると、**クラウド上の実体ファイルも完全に消去（30日以内）されます。**
*   **実務での使い所:** Databricks内で完結しているGold層の集計テーブルや、Silver層の中間テーブル等は全てこれにします。パフォーマンスも最高にチューニングされます。

### 🔗 外部テーブル (External Tables) / 外部ロケーション
*   **定義:** データの実体はユーザー企業が運用する既存のS3バケット上にあり、UCには「そのS3のメタデータ定義とアクセス権限（External Location）」だけを登録するテーブル。
*   **特徴:** `DROP TABLE` を実行しても、UCの登録が消えるだけで**S3上の実体ファイルはそのまま残ります。**
*   **実務での使い所:** Bronze層の入り口（他システムからS3に直接ファイルが置かれる場所）や、Databricks以外のツール（SnowflakeやRedshift等）からも同時に同じデータを参照したい場合は必ずこれを使います。

## 4. 動的アクセスコントロールによる行・列レベルのセキュリティ
UCを使えば、単に「このテーブルを見せる・見せない」という大雑把な `GRANT SELECT` だけでなく、SQL関数を使った「ダイナミック（動的）なビュー」が作れます。

> [!TIP]
> **実務での強力なユースケース**
> 1つの `sales_master` テーブルを全社に公開します。
> しかし、クエリを実行したユーザー（ログインしているメールアドレス等）の所属を見て、**「人事部の人間がクエリを実行した場合だけ、給与カラムを `***` にマスキングする」「A支店の店長がクエリを実行した時は、自動的にWHERE句に `branch = A` が挿入され、他店のデータが消えるようにする」**といった高度な制御（行レベル・列レベルセキュリティ）が、たった一つのテーブル（ビュー）の定義だけで実現できます。
