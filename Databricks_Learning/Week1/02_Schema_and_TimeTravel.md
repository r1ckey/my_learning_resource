## 1. スキーマ・エンフォースメント（強制）と基盤の防御

データレイク運用が失敗する最大の原因は、「どんな形式のデータでもとりあえず保存できてしまう」という自由度の高さにあります。上流のWebシステムが突然カラム名を変えたり、テストデータ（「hoge」といった無意味なカラム）を誤って混入させたりすると、それを読み込む下流のBIツールや集計バッチが一斉にクラッシュします。

Delta Lakeはデフォルト設定において、「スキーマ・エンフォースメント（強制）」が厳密に働きます。
*   既存のテーブル定義に存在しない**新しいカラム**を追加しようとすると直ちにエラーになります。
*   既存のカラムの**データ型が異なる**（例：INTで定義された列に、STRINGの文字データが入ってくる）場合もエラー弾きを行います。
*   カラム名の大文字・小文字はデフォルトでは区別されません（Case-Insensitive）。

### 実務での判断：「エラーで止める」べきか、「スキーマの自動拡張を許容する」べきか
「上流で1列新機能データが増えるたびにETLジョブが落ちて、深夜のオンコールでエンジニアが叩き起こされる」のは避けなければいけません。そのため、"追加"の列（データの拡張）をシステム的に許容する場合は、スキーマを進化（エボリューション）させます。

```python
# スキーマの進化（新しいカラムの自動追加）を許可する書き込み
df.write \
  .format("delta") \
  .mode("append") \
  .option("mergeSchema", "true") \ # これがスキーマ進化を許容するスイッチ
  .saveAsTable("my_table")
```

> [!WARNING]
> **実務環境の層に応じたベストプラクティス**
> - **Bronze層（生データ）**: 源泉データをそのまま保持するため、意図しないデータ欠損を防ぐ目的で、どんな列が増えても許容する（`mergeSchema = true` または Auto Loaderにおける該当オプション）設定にするのが実務のデファクトです。
> - **Silver層 / Gold層**: ここではデータ品質の「厳密さ（SSOT）」が求められます。`mergeSchema`は基本オフにし、集計用のスキーマ変更が必要な場合は明示的に `ALTER TABLE` などのSQLコマンドでエンジニアが構造を手動拡張すべきです。

## 2. タイムトラベル（Time Travel）によるデータ復旧と監査

前章（01_Architecture）で「Delta Lakeは既存のParquetファイルを直接上書きせず、変更のたびに新しいParquetファイルを作り、_delta_logに新バージョンとしてJSON記録する」と説明しました。
これはつまり、**「古いバージョンのParquetファイルも、クラウドストレージにそのまま残っている」**ということを意味します。これこそが「タイムトラベル機能」の仕組みそのものです。

### 最新の企業実務での活用シナリオ

1. **ヒューマンエラーの取り消し・DR対策 (Disaster Recovery)**
   うっかり本番環境で `DELETE FROM target_table` と WHERE 句なしで全件削除してしまった場合でも、バックアップから何時間もかけて戻す必要はなく、一瞬で元に戻す（RESTORE）ことができます。
   ```sql
   -- 過去の履歴一覧テーブルを確認し、安全だったバージョン番号(version)を特定
   DESCRIBE HISTORY target_table;

   -- バージョン「10」が安全だった場合、まるごとテーブルの状態をそこへ巻き戻す
   RESTORE TABLE target_table TO VERSION AS OF 10;
   ```

2. **機械学習モデルの再現性確保 (Reproducibility)**
   昨日「95%」の精度が出た機械学習モデルが、今日最新データで学習させたら「60%」に落ちたとします。実務では「アルゴリズムの設定が悪いのか、データそのものの分布が変わったのか」を切り分けるために、設定に問題がないことを証明するため過去時点のピクセルパーフェクトなデータでモデルを再評価する必要があります。
   ```python
   # "2024-03-25" 時点にテーブルが存在していた状態のデータフレームを正確に抽出してモデルに読ませる
   df_ml = spark.read.format("delta").option("timestampAsOf", "2024-03-25 00:00:00").table("features_table")
   ```

3. **セキュリティ監査とコンプライアンス**
   「いつ、どのジョブ（あるいはユーザーのメールアドレス）が、何件のレコードをUPDATEしたか」という操作情報が `DESCRIBE HISTORY` を通じて自動記録されているため、情報システム部門のセキュリティ監査要件をすぐに満たすことができます。
