# 第1週③：パフォーマンス最適化とファイル管理

データエンジニアの実務において、単にシステムからデータを取り込む仕組みを作るだけでなく、「いかに裏側のインフラコストを下げ、高速なユーザークエリ体験を提供するか（パフォーマンスチューニング）」が腕の見せ所です。
Delta Lakeにおける3つの最重要コマンド (`OPTIMIZE`, `ZORDER`, `VACUUM`) の仕組みと「実務でのスケジューリングの考え方」をまとめます。

## 1. 最重要課題：スモールファイル問題（The Small File Problem）
ストリーミング処理や、1時間ごとの細かなバッチ処理でひたすらデータを `APPEND` していくと、S3やADLS上に「数KB〜数MBの非常に小さなParquetファイル」が数万・数十万と爆発的に溜まっていきます。
SparkやDatabricksの分散処理エンジンの仕組み上、「1つの巨大な1GBのファイルを1回読み込む」より「1MBの小さなファイルを1000回読み込む」方が、ネットワーク通信やメタデータ展開のオーバーヘッドにより**圧倒的に（体感で数十倍）遅くなります**。

これを解決するのが `OPTIMIZE` コマンド（ファイル結合・コンパクション）です。

```sql
-- テーブル内の小さなファイルをかき集め、読み込みに最適なサイズ（約1GB）のファイルに結合・再編成する
OPTIMIZE target_table;
```

## 2. Z-Order クラスタリングによる異次元の高速化 (Data Skipping)
あるビジネス部門から「全期間のデータはいらない。特定の『最新月のデータ』と『特定店舗』しか見ないのに、ダッシュボードの表示が遅すぎる」と苦情がきたとします。
従来のリレーショナルデータベース(DWH)では、こうした検索用に B-Tree Index（インデックス）を張りますが、データレイクなどのビッグデータ基盤ではインデックスファイル自体の容量が巨大になりすぎて検索が破綻します。

代わりにDelta Lakeで使われるのが **Z-Order** クラスタリングです。
これは、指定したカラム（例：日付や店舗ID）の「似た値同士が、物理的に同じParquetファイルに入るように」データを多次元的にソートして保存し直す技術です。
これを行うと、検索時に「ああ、探している店舗のデータはファイル名AとCにしか入っていないな」と瞬時に判断し、残りの数千個のファイルを読み飛ばします（Data Skipping）。結果としてクラウドのI/Oコストが劇的に削減されクエリが爆速になります。

> [!IMPORTANT]
> **実務でのZ-Order判断基準とアンチパターン**
> - **最適な対象**: アナリストのSQLクエリの `WHERE` 句や `GROUP BY` で非常によく使われるカラム（最大で1〜3個程度が推奨）。例：`tenant_id`, `date`, `device_id`。
> - **NG（アンチパターン）**: カーディナリティが極端に高い（ユニークな値が多すぎる）列（各行ごとのランダムなUUIDなど）を指定しても、Z-Orderの並べ替えによるグルーピング恩恵を受けにくく、最適化計算時間が無駄にかかるだけになります。

## 3. VACUUM（ゴミ箱を空にする）とコンプライアンス管理
「タイムトラベルができる（＝古いParquetファイルが残っている）」ということは、最適化のために `OPTIMIZE` や `UPDATE` をくり返せばくり返すほど、使われなくなった古いファイルが無限にストレージに残り続ける（＝クラウドストレージ費用の右肩上がりの増大）ことを意味します。
この不要になった古い物理ファイルをクラウド上から「完全に」手放すのが `VACUUM` コマンドです。

```sql
-- デフォルトでは、7日より古いファイル（かつ最新テーブルの構成に関与しないゴミファイル）のみ削除
VACUUM target_table;
```

> [!CAUTION]
> **実務で直面するVACUUM後の注意点とGDPR要件**
> `VACUUM` を実行してクラウドから物理削除されたファイルには、もう**絶対にタイムトラベル（復元）することはできません。**
> GDPR（忘れられる権利）などのコンプライアンス要件で、「ユーザーからアカウント削除依頼が来たら、物理データベースからも30日以内に完全に消去しなければならない」という厳しいルールがある場合、レコードを `DELETE` 実行した後、その該当ファイルがいち早く古くなるのを待って `VACUUM` を高頻度で定期実行するアーキテクチャ設計が必須となります。

## ⚙️ 現場でのメンテナンス運用スケジュール（推奨アーキテクチャ）
ベストプラクティスとして、商用環境ではDatabricks Workflows（ジョブ）を使って以下のようなスケジューリングを組みます。
*   **毎日深夜 (もしくは数時間ごと)**: `OPTIMIZE my_table ZORDER BY (検索キー)` （新しく追加された乱雑なデータを整理）
*   **週末に1回 (ストレージ最適化)**: `VACUUM my_table RETAIN 168 HOURS`（デフォルトの7日間=168時間を過ぎた不要ファイルを一掃して従量課金コストを落とす）
