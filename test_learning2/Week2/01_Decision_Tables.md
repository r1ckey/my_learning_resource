# Test Week 2②：If文の迷宮を可視化する「デシジョンテーブル」

システム開発において、プログラマーの頭の容量（一時記憶）を最も簡単にぶっ壊し、大量のバグを生み出す原因があります。
それが **「複雑なビジネスルール（If文とフラグの嵐）」** です。

**【顧客からの要件（仕様書）】**
「うちのECサイトの割引ルールです。
・プレミアム会員なら「送料無料」です。
・ただし、プレミアム会員でも「冷蔵商品」が含まれている場合は、送料がかかります。
・一般会員は送料がかかります。
・しかし、一般会員でも「キャンペーンコード」を入力したら送料無料になります。
・でも「冷蔵商品」が含まれていた場合は、誰が何を入力しようと絶対に送料はかかります。」

これをプログラマーが `if (isPremium && !hasFridge) || ... ` などとコードを書き、テスターが「とりあえず色々組み合わせてみるか…」とフィーリングでテストした場合、**必ず「特定の組み合わせ」のテスト漏れが発生し、リリース後にクレームの嵐になります。**

## 1. デシジョンテーブル（決定表）の書き方

全ての複雑な条件を、頭の中ではなく「Excelのマトリクス（表）」に書き起こし、抜け漏れを可視化する最強の技法が **デシジョンテーブル (Decision Table)** です。

1.  **条件 (Conditions) を左上に書き出す:** 今回のユーザーの入力要素。
2.  **結果・動作 (Actions) を左下に書き出す:** システムが出すべき正しい答え。
3.  **右側に「Y / N (Yes / No)」の組み合わせを全て（2のn乗）埋め込む。**

| 条件・動作 (Conditions / Actions) | ケース1 | ケース2 | ケース3 | ケース4 | ケース5 | ケース6 | ケース7 | ケース8 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| **[条件] プレミアム会員か？** | Y | Y | Y | Y | N | N | N | N |
| **[条件] キャンペーンコードあり？** | Y | Y | N | N | Y | Y | N | N |
| **[条件] 冷蔵商品が含まれるか？**| Y | N | Y | N | Y | N | Y | N |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| **[結果] 送料無料** | | ◯ | | ◯ | | ◯ | | |
| **[結果] 送料発生（有料）** | ◯ | | ◯ | | ◯ | | ◯ | ◯ |

## 2. デシジョンテーブルが持つ【魔法の力】

この表を書き起こした瞬間、あなた（QAアーキテクト）は2つの圧倒的な力を手に入れます。

### ① テスト要件の「完全網羅」の証明
「私は、この8通りのテストを全て行いました。これにより、このECサイトの割引ロジックにおいて発生しうる**全ての組み合わせバグが100%存在しないことを数学的に証明（保証）します**」と言い切ることができます。

### ② 仕様の「矛盾（バグ）」の事前発見 (シフトレフト)
テストを実施する前（要件を書いている時）に、表をじっと見ていると気づきます。

*   **アーキテクト:** 「ん？ ケース1 と ケース2を見てください。プレミアム会員であれば、キャンペーンコードがあろうが(Y) なかろうが(N) 、結果（送料無料かどうか）には**全く影響を与えていない（どうでもいい条件になっている）**ぞ？」
*   **顧客（仕様元）への確認:** 「お客さん、プレミアム会員の人が間違えてキャンペーンコードを入力した時、どうします？『あなたは既に送料無料ですが、コードを使うとさらに割引にしますか？』などの別のアクション（仕様）が必要じゃないですか？今の仕様書には書いてありませんよ（仕様漏れの発見）！」

> [!TIP]
> **🚀 頭の良いテーブル圧縮（Don't Care の ` - `）**
> 今回の「冷蔵商品が含まれる＝Y」の場合、上の2つの条件が何であれ『絶対に送料が発生(Y)』しますよね。
> 実務のデシジョンテーブルでは、この「上（プレミアム）の条件がYでもNでも結果が全く変わらない」というセルの部分を **` - ` (ハイフン：どうでもいい / Don't Care)** と書き換えて、無駄な列（テストケース）を圧縮（間引き）してテスト工数を削減するテクニックを多用します。
