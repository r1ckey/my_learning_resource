# Test Week 4②：プロの直感「エラー推測」と「探索的テスト」

前章まで、Excelを使って「同値分割」「デシジョンテーブル」「状態遷移図」という【ガチガチの論理的・数学的な型（網羅）】を作ってきました。これを「仕様ベースのテスト設計」と呼びます。

しかし、現場で何千回もテストを行ってきたプロのQAエンジニア（テスター）は、それだけでは満足しません。最後は自らの「経験」と「直感」という、AIにも作れない人間固有の泥臭い武器を抜きます。

## 1. エラー推測 (Error Guessing) の極意

仕様書には「名前を入力して保存してください」としか書いていない。
論理学（同値分割）でいえば、「"山田太郎"（有効な値）」と、「長すぎる文字（境界値エラー）」をテストすれば設計完了です。

しかし、プロのQAエンジニアは仕様書を無視して、次のような**「システムを破壊する魔法の呪文（経験則）」**を思い描き、テストのExcel（香盤）の一番端っこに、ニヤリと笑いながらコッソリ書き加えます。

*   **プロの第六感（過去に地獄を見た経験のリスト）:**
    1.  **「NULL文字/空白文字」:** 「あ〜あ、このプログラマー、絶対スペースキー（全角スペース）だけの名前を入力された時のエラー処理書いてないだろうな。よし、試してやろう（クラッシュ）」
    2.  **「特殊な文字（サロゲートペア / 絵文字）」:** 「昔のMySQLデータベース使ってるらしいから、名前に 𩸽 (ほっけ) みたいな文字や、🍔 (ハンバーガー) の絵文字を入れたら、DB保存時に文字化けしたり500サーバーエラーで爆発するはずだ！」
    3.  **「うるう年問題（2月29日）」:** 「仕様は『1年後の日付を表示する』か。よし、入力日を『2024年の2月29日』にしてシステム時間を進めてみよう。1年後の2025年2月29日は実在しないから、絶対カレンダー関数がバグってカレンダーアプリが落ちるぞ。」
    4.  **「0除算 (Zero Division)」:** 「売上 ÷ ユーザー数 のKPIダッシュボードか。ユーザー数が0人の時、このSQLは『ゼロで割り算した』エラーで死ぬはずだ。」

これら「仕様書に書いていない、エンジニア特有の過去のバグ経験のデータベース」から引いてくるのが、エラー推測という強力な補完技法です。

## 2. 決戦場：探索的テスト (Exploratory Testing)

仕様ベースで、ガチガチの網羅は全て済ませた。エラー推測もやった。
それでも「システムの奥底にある真のバグ」は、実際に人間が適当にアプリを使い回している（サルみたいに触っている）時に、偶然の組み合わせでポロリとこぼれ落ちてくるものです。

これを「適当なテスト（モンキーテスト）」ではなく、「知的探求」に昇華させたのが**探索的テスト**です。

### 🧭 探索的テストの特徴と進め方
*   **特徴:** 事前にExcelにテスト項目（手順）を書きません。**【テストの設計（考える）】と【テストの実行（操作する）】を完全に同時に（並行して）行います。**
*   **流れ（マインド）:**
    1.  「よし、まずログイン画面を開こう（実行）。あれ？この『パスワードを忘れた方』のリンクを踏んでから、ブラウザの『戻る』ボタンを2連打すると、なんかセッション（Cookie）が切れて、未ログイン状態でマイページの中身が一瞬見えちゃってないか？（発見・推測）」
    2.  「じゃあ、このバグが本当か確かめるために、別のブラウザを開いて同じ手順をもう一回やってみよう（即席のテスト設計と実行）」
    3.  「本当だ！バグだ！」（バグ報告チケットの起票）

> [!CAUTION]
> **🚨 探索的テストの『免罪符（言い訳）』利用の禁止**
> 探索的テストは、非常に効率的にクリティカルなバグ（使い勝手やUX、複数画面にまたがるタイミングバグ等）を見つけることができます。
> **しかし、「面倒くさいから、Excelにテストケース群の設計を書かずに、探索的テスト（という名の適当にクリックするだけの作業）だけでリリースを済ませちゃおう！」というのは、世界最悪のアンチパターン（犯罪行為）です。**
> 
> 探索的テストは、「同値分割」や「デシジョンテーブル」を使って、システムの網羅性（カバレッジ品質）を数学的に100%保証した【その後の仕上げ・追加の遊びの時間】にのみ許される、トッププロのための最終奥義なのです。
