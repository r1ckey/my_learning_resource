# Test Week 1②：最小の手数で最大を仕留める「同値分割法」と「境界値分析」

**「1歳 から 100歳 までの数字（半角）を入力してください」**
あなたがこの入力フォーム（テキストボックス）のテストを任されました。

100回、すべての数字を打ち込みますか？
「馬鹿らしい。じゃあ、代表して『5』と『50』と『99』の3つを入れて、エラーが出なければOKでいいじゃん（フィーリング）」

どちらも三流のテストです。プロの「ブラックボックス設計技法」の基本である2つの強力な武器を使います。

## 1. 同値分割法 (Equivalence Partitioning)

「ある条件（プログラム）から見て、**『どれを入力しても全く同じ結果（同じ扱い）になるデータの巨大なグループ』**をひとまとめ（同値クラス）として捉え、その中から**『代表選手をたった1人だけ』選んでテストすれば、そのグループ全員をテストしたのと同じである**」という強力な理屈です。

*   **グループ分け:**
    *   `[無効なグループA (以上)]` : 101, 102, 1000... (エラーになるはず)
    *   `[有効なグループB (正常)]` : 1, 2, 50, 99, 100... (OKになるはず)
    *   `[無効なグループC (以下)]` : 0, -1, -50... (エラーになるはず)

*   **テストケース（代表選手を一人選ぶ）:**
    *   ケース1 (Aから選出): `150`を入力 ➡️ 「エラー: 100以下にしてください」
    *   ケース2 (Bから選出): `35` を入力 ➡️ 「正常: OK」
    *   ケース3 (Cから選出): `-10`を入力 ➡️ 「エラー: 1以上にしてください」

これで、「最低限のシステム要件（正・負の世界の網羅）」がたった『3件のテスト』で完了しました。

## 2. 境界値分析 (Boundary Value Analysis / BVA)

同値分割法で代表選手（`35`などの中央値）を選んでテストしたあなたに、シニアエンジニアがこう言います。
**「君のテストは甘すぎる。プログラマーのバグの99%は『真ん中』ではなく『一番端っこ（境界線）』に潜んでいるんだ。」**

プログラマーがコードを書く時、最も間違えやすいのが以下の記号です（`>=` と `>` の打ち間違い）。
```javascript
// プログラマーの単純なミス（本当は >= 1 と書くべきだった）
if (age > 1 && age <= 100) { "OK" } else { "ERROR" }
```
このバグコードは、先ほどの同値分割（代表の `35`）では普通に通り抜けて「正常終了（テスト合格）」してしまいます！

*   **境界値（端の数字）を執拗に狙え:**
    グループの「一番端っこ（境界）」と、その「すぐ外側の値（1ミリはみ出た値）」のセット（2値テスト）を追加します。

*   **今回の最強のテストケース抽出:**
    *   下限の境界: `0` (エラー) と `1` (正常)
    *   上限の境界: `100` (正常) と `101` (エラー)

もし先ほどのバグコード（`age > 1`）だった場合、あなたが境界値の **`1`** を入力した瞬間に、システムは「エラー（1より大きい必要があるため弾かれる）」という予想外の挙動をし、見事にバグを叩き出すことができます！

> [!IMPORTANT]
> **🚀 現場のテスト仕様書（Excel）のマスト項目**
> 
> あなたがこれからテスト仕様書（香盤）を作る時は、どんな入力欄であっても、どんなSQLの抽出テストであっても、**必ず「同値分割の中央値付近」と「境界値ギリギリの正常・異常」をセットで書き起こしてください。**
> Excelの項目に「1（下限の正常）」「0（下限の異常）」「100（上限の正常）」「101（上限の異常）」と並んでいるのを見たクライアントやマネージャーは、「こいつのテスト設計、完璧だプロだ（一切の漏れがない）」と一発で信頼を寄せることになります。
