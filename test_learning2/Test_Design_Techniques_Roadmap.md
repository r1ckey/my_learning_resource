# Part 2：テスト要件抽出と「設計技法」の基本型 ロードマップ

「この入力フォームのテストをやっておいて」
と言われた時、素人のエンジニアは「適当に数字を入れてみて、保存ボタンを押して、エラーが出なければヨシ！」という**「思いつき（探索的）」**のテストをやって終了します。

しかし、プロのQA（品質保証）やトップレベルの開発者は違います。
**「この入力欄には、『論理的（数学的）に考えて、この3パターンの値だけ入れれば、無限のパターンを入れたことと同じ証明になる』」**という、確立された【テスト設計技法】を使います。

この技法を知らずにExcelにテストケースをダラダラ書くのは、プログラマーが「For文（ループ関数）を知らずに、コードを1万行コピペして書いている」のと同じくらい非常に恥ずかしく、非効率で、バグだらけの行為です。

1. **Week 1: 端っこを狙い撃つ「同値分割法」と「境界値分析」**
   * 「1から100まで入力できる年齢欄」。1〜100まで全部テストするのは馬鹿げています（全数テストは不可能）。ならば、どの「数字」を選んでテストするのが数学的に最小かつ最強のバグ発見率を誇るのか。
2. **Week 2: IF文の迷宮を可視化する「デシジョンテーブル」**
   * 「30歳以上で、かつVIP会員で、かつ平日なら、50%割引」。この複雑な複数フラグの組み合わせパターンのテストの抜け漏れ（仕様の欠陥）を、Excelの表だけで論理的に全て叩き出す技法。
3. **Week 3: 全ての動きを網羅する矢印「状態遷移テスト」**
   * 「ステータス（下書き→申請中→承認済）」に依存して動くアプリ（決済やワークフロー）において、ユーザーが発狂する『ありえない画面遷移バグ』をグラフから見つけ出す方法。
4. **Week 4: プロの勘を言葉にする「エラー推測」と「探索的テスト」**
   * 型を学んだ上で、最後はエンジニアとしての経験（第六感）。「あ、このデータベース、文字コードのバグが出やすそうだな」等、仕様書には載っていない異常系（探索の旅）のメソッド。
