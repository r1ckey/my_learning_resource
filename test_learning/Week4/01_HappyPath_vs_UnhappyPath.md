# Test Week 4：Happy Path (正常系) と Unhappy Path (異常系) の思想

あなたが「ログイン画面のテスト項目（Excel）」を任されたとしましょう。
初心者は必ず「正しいIDと、正しいパスワードを入れて、ログインできること（1件のテスト）」だけを書いて満足します。
この時、あなたは「テストエンジニア」ではなく、ただの「システムのお客さん（ユーザー）」です。

テストの本質は、ユーザーが絶対にやらない（あるいは間違えてやってしまう）**「不幸なルート（Unhappy Path）」をいかに大量に、悪意を持って洗い出せるか**に尽きます。

## 1. 正常系 (Happy Path / 正常ルート)

*   **定義:** システムが設計通りに、何も問題なく「理想的」に使われた時のルート。
*   **目的:** 「システムの一番売りとなるメインの機能（コアバリュー）」が、最低限ちゃんと提供できることを証明します。
*   **割合:** 全体のテストケースの **たった10%〜20%** に過ぎません。

## 2. 異常系・準正常系 (Unhappy Path / エッジケース)

*   **定義:** ユーザーの「誤入力（意図的・非意図的）」や、システムの「環境トラブル（DB切断等）」など、理想とはかけ離れたあらゆる悪条件のルート。
*   **目的:** アプリが「クラッシュして画面が真っ白にフリーズする」ことを防ぎ、代わりに**「パスワードが違いますよ」「現在ネットワークが混雑しています」という『安全で親切なエラーメッセージ』を出して、元の画面に綺麗に引き返させる（フェイルセーフ）**ことを証明します。
*   **割合:** 現場のテストケースの **約80%** は、このUnhappy Pathをいかに無限に想像し、網羅するかに費やされます。

> [!CAUTION]
> **🚨 異常系・準正常系を洗い出す「悪魔のチェックリスト」**
> 「入力欄（Input）」を見たら、プロのテスターは少なくとも以下の悪戯（バリデーション確認）を瞬時に思い浮かべます。
> 
> 1.  **空っぽ (Empty):** 何も入力せずに「送信」ボタンを押す（NULL検証）。
> 2.  **文字超過:** 「名前欄（MAX50文字）」に、わざと51文字を入れる。
> 3.  **異データ型（Type）:** 「年齢（数字）」の欄に、「あいうえお（文字）」を入れる。
> 4.  **特殊記号:** 予約語（SQLインジェクションになり得る `'` や `;` や `OR 1=1` など）を入れる。
> 5.  **環境破壊（物理エラー）:** 通信中（ロードくるくる中）に、いきなりスマホを「機内モード（圏外）」にする。あるいは「戻る」ボタンと「送信」ボタンを秒間10連打する。

### 📊 テストケースの比率（黄金比）

テスト設計の実務において、「1つの正常系」を見つけたら、それに付随する「3〜5個の異常系・準正常系」を必ずセットで洗い出し、香盤表（Excel）のリストに追加していくのが黄金ルールです。

*   `[正常系]` 正しいID・PWでログインすると、マイページに遷移すること。
*   `[準正常系]` 間違ったPWを入れると、「PWが違います」と赤字で表示されること。
*   `[異常系]` PWを3回連続ミスすると、「アカウントロック」がかかること。
*   `[異常系]` DBサーバーがダウンしている時に送信すると、「後でもう一度お試しください（HTTP 500等の画面が出ない）」と安全にエラー表示されること。

次週の **Part 2** では、この「異常系」を、勘やフェイリングではなく**「数学的・論理的に漏れなく網羅（リストアップ）する最強の設計技法（同値分割法・デシジョンテーブルなど）」**について深く学んでいきます。
