# Test Week 2：設計との対話「V字モデルとテストレベル」

システム開発（特にウォーターフォール開発）において、すべてのエンジニアが共有している**「いつ、何を、誰が、どうやってテストするのか」**の設計図が **V字モデル (V-Model)** です。

テストは最後にガッとまとめてやるものではありません。「左側（開発・要件）」の裏返しとして、「右側（テスト・検証）」が存在します。

## 1. V字モデルの構造（4つのテストレベル）

開発工程が下（詳細）に進んでいくのに対し、テスト工程は下から上（全体）へと登っていくため「V」の字になります。

### ① 要件定義 ⬅️対応➡️ UAT（ユーザー受け入れテスト）
*   **左側:** お客様が「こういう機能が欲しい」と決める。
*   **右側 (UAT: User Acceptance Testing):** 開発の本当に最後。お客様自身がシステムを触り、「私が欲しかったのはコレコレ！」と納得（受け入れ）するテスト。ここでバグが見つかると設計の根本からやり直しになり地獄を見ます。

### ② 基本設計 ⬅️対応➡️ ST（システムテスト / 総合テスト）
*   **左側:** 「画面はこういうデザインで、こういう機能一覧が必要」と決める。
*   **右側 (ST: System Testing):** すべてのプログラムが完成した後に、「システム全体の動作（画面のボタンからDBの保存まで）」が、基本設計書の通りに動くかをテストします。

### ③ 詳細設計 ⬅️対応➡️ IT（結合テスト / Integration Testing）
*   **左側:** 「この画面モジュールと、後ろのデータベースモジュールはこのAPIで通信する」と決める。
*   **右側 (IT):** それぞれ作られたバラバラの部品（関数やクラス）を「ガッチャンコ（結合）」して、ちゃんとデータが受け渡しできているか（通信エラーが出ないか）をテストします。

### ④ 実装（コーディング） ⬅️対応➡️ UT（単体テスト / Unit Testing）
*   **左右の最底辺 (UT):** V字の一番下。プログラマーがコード（関数単位、クラス単位）を書いた直後に、その**「1つの関数（部品）」が他から完全に孤立した状態で、期待通りの処理をするか**を確認します。

## 2. なぜ「テストレベル」を分けるのか？（現場の鉄則）

初心者のプログラマーは、「関数を書き終わったら、実際にブラウザの画面を開いて、ボタンをポチポチ押して（通しで・STで）テストすればいいじゃん」と考えがちです。

なぜ、面倒臭い「単体テスト（UT）用の専用コード」をわざわざ書いたり、分割してテストをするのでしょうか？

> [!IMPORTANT]
> **🌟 バグの【原因特定スピード】の違い**
> 
> *   **いきなり全体（ST）でテストして「エラー」画面が出た場合:**
>     「えっ？画面側のJavascriptのバグ？それともAPIの通信バグ？それともデータベースのSQLの構文エラー？どこでコケたのか全く分からない！」となり、原因の特定に数時間かかります。
> *   **単体テスト（UT）を徹底していた場合:**
>     UTの段階で「関数Aは正常」「関数Bは正常」「SQLは正常」というお墨付き（証明）がそれぞれ得られています。
>     その状態で結合（IT/ST）してエラーが出た場合、**「部品単体は完璧だから、絶対に関数Aと関数Bの『繋ぎ目（インターフェース・引数）』のどちらかで型がズレているだけだ！」**と、数秒で原因のありかを絞り込むことができるのです。
> 
> 「バグの発見」自体はどのレベルでも出来ますが、**「バグの原因究明（デバッグ）の速さ」と「他システムへの波及の防波堤」**のために、私たちは徹底して小さなレベル（UT）から順に積み上げてテスト（品質保証）をするのです。
