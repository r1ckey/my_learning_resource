# 認定対策 Week 1③：Git Folders (Repos) と CI/CDベストプラクティス

Databricks環境において、コード（ノートブックやファイル）をGitHub等のバージョン管理システムと連携させる機能が **Databricks Git Folders（旧称：Repos）** です。
Associate試験では、本格的なCI/CDパイプラインを組むコマンドは出ませんが、「運用上の**絶対にやってはいけないこと**（アンチパターン）」が執拗に問われます。

## 1. Databricks Git Folders の基本概念

*   Databricks画面の左メニューにある「Workspace」とは別の、「Git専用の領域」です。
*   ユーザー単位（個人のフォルダ）、または本番環境用ディレクトリに、GitHubのリポジトリをそのまま同期（Clone）して展開できます。
*   Git Folders内でノートブックのコードを書き換えると、Gitの「Commit」や「Push」といった操作がUI上からボタンポチポチで簡単に行なえます。


## 2. 本番環境（Production）へのリリースの「正解」

試験で最も頻出するのが、**「エンジニアが新しいETLのコードを本番環境にデプロイ（反映）したい場合、最も安全で推奨される手順はどれか」** という問題です。

### 🚫 [絶対の不正解（Exam Trap）]
1. 「本番環境の `main` ブランチに紐付いているノートブックを、直接開いてコードを書き換える。」
   *   理由：Gitの履歴を無視した無法地帯となり、コンフリクト（競合）や先祖返りを引き起こします。
2. 「個人環境で作ったノートブックファイルを、DBFS(分散ファイルシステム)のフォルダにそのままエクスポート/インポートして上書きする。」
   *   理由：バージョン管理がなされておらず、誰がいつ変更したか監査できなくなります。

### ✅ [絶対の正解（Best Practice）]
1. エンジニアは、**自分個人のGit Folder** 内で、`main` から別の **開発用ブランチ（dev等）を切って** 改修を行う。
2. 完了したら、GitHubにPushし、チームメンバーにGitHub上で「プルリクエスト (Pull Request)」を出してレビューしてもらう。
3. コードが承認されて GitHub側の `main` にマージされたら、
4. Databricks API や CI/CDツール（GitHub Actionsなど）を使って、**プロダクション用のGit Folderに「GitHubの最新のmainをPullせよ」と命令を出して同期させる。**

> [!IMPORTANT]
> **🚨 Exam Focus (キーワード回収)**
> 試験で「本番環境の更新方法」を問われたら、正解の選択肢には必ず以下のキーワードが含まれています。
> **「Pull Request (プルリク) を作成する」**
> **「Databricks API / CI/CD pipeline を使用して本番フォルダをアップデートする」**
> （※「手動で直接コードを変更する」系の選択肢はすべて罠です。）

## 3. シークレット（パスワード等の機密情報）の管理
Git Foldersを使う際、S3のキーやデータベースのパスワード等をノートブックに直書きしてコミットすると、GitHubにパスワードが公開されて大問題になります。

*   **[正解の対応]**: **Databricks Secrets（シークレット）機能** を使用する。
*   ノートブック内からは `dbutils.secrets.get(scope="my_scope", key="db_password")` という見えない箱(API)を通してパスワードを呼び出します。
*   試験では、「機密情報はどこに保存すべきか？」と聞かれたら即座に **Databricks Secrets（またはAzure Key Vault）** と答えてください。
