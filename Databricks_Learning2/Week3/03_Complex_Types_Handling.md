# 認定対策 Week 3③：複雑型（配列・JSON）と高階関数の攻略

実務における「非構造化データ（ネストされたJSON）」の処理は、Associate試験の終盤で最も頭を悩ませる領域です。
特に配列（Array型）の処理については、SQLにおける**「高階関数 (Higher-Order Functions)」**と呼ばれる独特の構文が好んで出題されます。

## 1. `explode`（爆発・展開）関数の絶対定義

JSONログなどで、「1つのカラムのレコード（例えばセルの中）に、配列データのリストが丸ごと書き込まれている」ことがあります。

```json
{"user_id": 1, "purchases": ["apple", "banana", "orange"]}
```

これを、分析しやすいように「1行＝1つの果物」という縦長のテーブルに変換（フラット化）する関数が **`explode`** です。

| user_id | purchases |
| :--- | :--- |
| 1 | apple |
| 1 | banana |
| 1 | orange |

> [!WARNING]
> **🚨 Exam Trap (`explode_outer` との違い)**
> 上記の `user_id = 2` の人がいて、その人は何も買っていないため配列が空（`[]`）や `NULL` だったとします。
> 
> *   通常の `explode("purchases")` をかけると：
>     **副作用（Trap）:** その `user_id = 2` の行ごと完全に**消滅（Drop）してしまいます**。
> *   消滅させず、`apple` などの代わりに `NULL` を入れてでも、元の親行（ユーザー情報）を維持したい場合は：
>     **[絶対の正解]**: **`explode_outer("purchases")`** を使います。
>     （※SQLの「LEFT OUTER JOIN」と発想は完全に同じです）。

## 2. Array型専用の高階関数（Higher-Order Functions）

試験では「Spark SQL環境で、配列（Array）の中身を順番にループ処理（イテレーション）したい」という問題が出ます。
Pythonの `for x in list:` のようなやり方ではなく、**SQL構文の中に組み込まれたラムダ式（`->`）** を使って処理するのがDatabricksの特徴です。

### 最頻出①： `FILTER` 関数
配列の中から、**特定の条件を満たす要素だけを残して新しい配列を作る**。

```sql
-- 構文 (試験でこのラムダ式 `t -> t.xxxx` が穴埋めで出ます)
SELECT FILTER(
    array(1, 2, 3, -1, 5),   -- 対象の配列
    t -> t > 0               -- t (一つ一つの要素) が 0 より大きいものだけを残す条件
);
-- 結果: [1, 2, 3, 5]
```

### 最頻出②： `TRANSFORM` 関数
配列の中の要素一つ一つに対して、**特定の計算や文字列加工を行い、全く別の新しい配列に変換する**。
※ Pythonの `map()` 関数と全く同じです。

```sql
SELECT TRANSFORM(
    array(1, 2, 3), 
    t -> t * 10 
);
-- 結果: [10, 20, 30]
```

> [!IMPORTANT]
> **🚨 Exam Focus (キーワード回収ゲーム)**
> 問題文で「SQL（Spark SQL）を使って、**配列 (Array) の中の要素を複雑に操作（抽出・変換）したい** が、UDF（ユーザー定義関数）を使うと遅くなるので避けたい。Sparkの組み込み関数として何を使うべきか？」と問われます。
> 
> **正答率100%の答え**: **Higher-Order Functions（高階関数）** または **FILTER / TRANSFORM** です。「UDFの代わりに使う」という文脈で覚えてください。
