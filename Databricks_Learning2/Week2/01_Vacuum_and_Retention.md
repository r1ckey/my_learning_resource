# 認定対策 Week 2①：VACUUMの「7日間ルール」とタイムトラベルの限界

Delta Lakeの最大の強みである「Time Travel（過去データの復旧）」と、ストレージコストを抑えるための「VACUUM（不要ファイルの削除）」コマンド。
この2つはトレードオフの関係にあり、試験ではその**「閾値（保持期間）」**を中心としたひっかけ問題が大量に出題されます。

## 1. タイムトラベル（Time Travel）の基本と上限
Delta Lakeは、データの一部が削除・変更されるたびに、古いバージョンのParquetファイルを消さずに残しておきます。だからこそ、間違えてデータを全消ししても `RESTORE` コマンドで一瞬で復活できます。

*   **RESTORE 構文 (試験で問われます)**
    *   `RESTORE TABLE test_table TO VERSION AS OF 10;`
    *   `RESTORE TABLE test_table TO TIMESTAMP AS OF '2024-03-01 10:00:00';`
*   **デフォルトの履歴の上限**: トランザクションログ（`_delta_log` ディレクトリ内のjsonファイル群）のデフォルト保存期間は **30日間** です。これより前の状態にタイムトラベルしようとするとエラーになります。

## 2. VACUUM（バキューム）コマンドとは？
タイムトラベルのために古いファイルを残し続けると、Amazon S3のストレージ費用が毎月天文学的に膨張します。
そこで、**「トランザクションログからは最新バージョンから外れていて、誰も参照していない古い実体データ（Parquetファイル）を、クラウドから物理的に完全削除して容量を空ける」** のが `VACUUM` コマンドです。

```sql
-- テーブルの古い不要ファイルを削除する基本コマンド
VACUUM events_table;
```

## 3. 【超重要】VACUUMの「7日間ルール」の罠

> [!WARNING]
> **🚨 Exam Trap (超頻出の引っかけ問題)**
> **[ルール]**: `VACUUM` をオプションなしで実行すると、「**過去7日間（168時間）** 以内に変更された古いファイル」は、安全のために**絶対に消されずに保護されます**。(※デフォルトの `retention period` が7日間であるため)。
> 
> つまり、「間違えて1時間前にデータを消しちゃった！さらに今の状態でVACUUMをかけちゃった！もう1時間前にタイムトラベルして復元することは不可能か？」という問題が出た場合、
> **[絶対の正解]:** 「1時間前は7日間の保護期間内なので、VACUUMによってファイルは消されておらず、**タイムトラベルで復元可能**」です！

## 4. RETAIN 0 HOURS の意地悪問題

試験では、「ストレージを極限まで節約したいデータエンジニアが、猶予期間なし（0時間前までの変更履歴を全て即座に削除）でVACUUMをかけようとした」というケースが出題されます。

```sql
-- 0時間前(今)より前の古い履歴をすべて削除したいという意図
VACUUM events_table RETAIN 0 HOURS;
```

> [!IMPORTANT]
> **🚨 Exam Focus (エラーの回避方法)**
> 上記のコマンドを実行するとどうなるでしょうか？
> **[絶対の正解]:** **デフォルトではエラーが発生してコマンドは失敗します。**
> Databricksは、今まさに実行中の長時間クエリ（3時間かかるJoinなど）が参照しているファイルを誤って消してしまわないよう、防波堤を張っているからです。
> もし本当に強制的に7日間未満（0時間など）でVACUUMしたい場合は、セッションの設定で
> `SET spark.databricks.delta.retentionDurationCheck.enabled = false;`
> と明示的にストッパーを無効化しなければならない、というのが試験で要求される知識です。
