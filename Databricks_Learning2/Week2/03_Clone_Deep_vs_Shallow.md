# 認定対策 Week 2③：テーブルクローン (Deep vs Shallow) の物理構造

Databricks環境において、テーブルを複製（コピー）するコマンドが `CLONE` です。全く同じデータを持つテーブルを作る際に、**Cloud Storage上の物理ファイル（Parquetファイル）がどう処理されるのか**という仕組みが、Associate試験で極めて高い確率で出題されます。

「DEEP CLONE」と「SHALLOW CLONE」の2つの特性と違いを脳に焼き付けてください。

## 1. DEEP CLONE (ディープ・クローン) の挙動

```sql
-- 構文
CREATE TABLE dev_table DEEP CLONE prod_table;
```

*   **内部の動き**:
    1.  元の `prod_table` のメタデータ（スキーマや設定）をコピーします。
    2.  背後にあるS3等のクラウドストレージ上の **全てのParquetデータファイルも、別の場所に物理的にコピーして複製します。**
*   **特徴**:
    *   データのコピー処理が発生するため、大きなテーブルだと**完了までに数十分〜数時間の非常に長い時間がかかります**。
    *   コピーした分だけ、**ストレージ容量（クラウド課金）が2倍に増えます**。
    *   複製後は、元の `prod_table` と完全に独立しています。**もし元の `prod_table` を全消去（DropやVacuum）したとしても、クローンした `dev_table` は無傷で残ります。**（これが重要）

## 2. SHALLOW CLONE (シャロー・クローン) の挙動

```sql
-- 構文
CREATE TABLE test_table SHALLOW CLONE prod_table;
```

*   **内部の動き**:
    1.  元の `prod_table` のメタデータ（トランザクションログ）だけをコピーします。
    2.  **実体データであるParquetファイルは一切コピー（複製）しません。**クローンされた側のテーブルは、ずっと「大元の prod_table の実体データ」へリンクを張って参照するだけです。
*   **特徴**:
    *   ファイルをコピーしないため、何テラバイトの巨大テーブルであっても **数秒で一瞬で完了** します。
    *   実体が増えないため、**追加のストレージコストはゼロ** です。（※その後の更新差分だけが容量を食います）。
    *   テスト用に一瞬で本番データのダミー環境を作りたいときに最適です。

> [!WARNING]
> **🚨 Exam Trap (Shallow Clone の最悪の弱点)**
> 実務でも「シャロークローンは早くて無料だからこれでいいじゃん」と考えがちですが、試験では以下の致命的な弱点が必ず問われます。
> 
> **[問題設定]**: 「エンジニアが `prodA` テーブルから **SHALLOW CLONE** を使って `testB` テーブルを作りました（目的は検証）。数日後、本番のストレージを節約するために、大元の `prodA` テーブルに対して `VACUUM` コマンドを実行し、不要なファイルとして大元の実体データをクラウドから消去しました。さて、`testB` に対して `SELECT *` を実行するとどうなるでしょうか？」
> 
> **[絶対の正解]**: **`testB` も巻き込まれて壊れ（File Not Found Exception）、エラーになり読み込めなくなる。**
> （理由：Shallow Clone は大元のファイルを参照しているだけなので、大元ファイルの足場がVACUUM等で崩されると、自分自身も道連れで崩壊します）。
> 
> 「一時的なテストならShallow」、「元が消えてもずっと残したい完全なコピーならDeep」という判断基準がテストで試されます。
