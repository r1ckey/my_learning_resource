# Azure Week 1①：Entra ID と RBAC（最強の権限管理）

クラウドで一番最初に学ぶべきであり、最も恐ろしい事故（情報漏洩）に繋がるのが「アイデンティティ（認証）」と「アクセス制御（認可）」です。
AZ-104では、この2つの概念の違いを「絶対に間違えないレベル」まで仕上げる必要があります。

## 1. Microsoft Entra ID (旧: Azure AD)
会社のパソコンにログインする時に使うシステムのクラウド版です。「あなたは誰か？（パスワードは合っているか）」を **認証（Authentication）** します。

> [!WARNING]
> **🚨 Exam Trap (旧型Active Directoryとの違い)**
> 「社内にあるオンプレミスの『Windows Server Active Directory (AD DS)』を、そのままクラウドに持ってきて置き換えたものが Entra IDである。」
> ➡️ **[絶対の不正解（大嘘）です]**
> 
> *   **オンプレAD**: Kerberos認証などを使い、グループポリシー（GPO）でPCを管理する。
> *   **Entra ID**: OAuth 2.0 や SAML などのWeb標準認証を使ったフラットなディレクトリ。「GPOによるパソコンの制御」などの機能は一切持っていません（※GPOの代わりにIntuneを使います）。
> 
> *「GPOを使って管理したいからEntra IDを使う」という選択肢は必ず引っ掛け（バツ）になります！*

## 2. 権限管理の二重構造（絶対に混同しないこと！）

Azureを操作する時、私たちは「2つの世界」のどちらかの権限を使って生きています。この境界線が試験の最大の罠です。

### ① Microsoft Entra のロール (ディレクトリ権限)
*   **対象:** 「ユーザー作成」「パスワードリセット」「ライセンス管理」など、**人間（アカウント）に対する操作**。
*   **王様ロール:** `Global Administrator` (グローバル管理者)
*   ➡️ この王様であっても、**他人が作ったAzure上のVM（仮想マシン）を勝手に再起動したり、削除したりすることは（デフォルトでは）できません！**

### ② Azure RBAC (Azureリソース権限)
*   **対象:** 「VMの起動/停止」「SQL DBの設定変更」「VNetの作成」など、**Azure上のモノ（リソース）に対する操作**。
*   **王様ロール:** `Owner` (所有者) または `Contributor` (共同作成者)

> [!IMPORTANT]
> **🚀 Exam Focus (よくある出題)**
> **[問題]**: 「新しい社員が入社したためAzureのポータルで『新規ユーザー』を作成したい。また、そのユーザーに『VMの再起動』ができる権限を与えたい。あなたには何のロールが必要か？」
> 
> **[正解]**: 新規ユーザー作成のために「**User Administrator (Entraロール)**」が必要。かつ、VM操作権限を与えるために、そのVMに対する「**Owner (RBACロール)**」等が必要。
> （必ず2つの世界を別々に考えて回答してください）

## 3. RBACのスコープ（継承ルール）と「Deny」の強さ

RBACの権限は、マトリョーシカのように「大きな箱（上位）」に与えられた権限が、「小さな箱（下位）」へ自動的に全て**下りて（継承されて）**いきます。

*   **スコープ階層:** `管理グループ (最上位)` ➡️ `サブスクリプション` ➡️ `リソースグループ` ➡️ `個別のリソース(VM等)`

> [!CAUTION]
> **🚨 Exam Trap (Denyの最強ルール)**
> 「Aさんは、会社全体（上位）で『仮想マシンを作っていい(Allow)』という**Contributor権限**をもらっています」
> 「しかし、ある一つのリソースグループ（下位）だけ、『仮想マシン作成禁止(Deny)』の設定（後述のAzure Policy等による拒否）が割り当てられていました」
> 
> 結果、Aさんはその下位グループでVMを作れるか？
> ➡️ **[作れません！]**
> 
> Azureの権限は、**「どれだけ強いAllow（許可）を持っていても、たった1つの明示的なDeny（拒否）があった瞬間、すべてDenyが優先される」**という絶対ルールを持っています。試験で権限のコンフリクト（衝突）が出たら「Denyの勝ち」と覚えてください。
