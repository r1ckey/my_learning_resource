# Azure Week 3②：NSG（ファイアウォール）の優先順位アルゴリズム

VNetやSubnet（町内会）を作っただけでは、世界中のハッカーから無差別攻撃を受け放題になってしまいます。
AZ-104試験では、「どの通信を通し、どの通信をブロックするか」という検問所、**Network Security Group (NSG: ネットワークセキュリティグループ)** のルール判定ロジックが確実に問われます。

## 1. NSG とは何か？
Azureにおける無料の「分散型の簡易ファイアウォール」です。以下の2箇所に設定（アタッチ）して使います。
1.  **Subnet（サブネット全体）の入り口**
2.  **VMのNIC（ネットワークカード・個別のPC）の入り口**

> [!TIP]
> **🚀 アーキテクトの設計：実務での使い方**
> 個別のVMのNICにNSGを100個つけると管理が破綻してしまいます。基本は**Subnetレベルに1つのNSG**をドンと被せ、そこでまとめてアクセス制御（例：WebサーバーのSubnetにはPort 80と443だけ通す）を行うのがベストプラクティスです。

## 2. 優先順位（Priority）の「数字バトル」

NSGのルール（許可・拒否）を設定する際、必ず **「100 〜 4096 までの優先順位（Priority）の数字」** を入力させられます。

*   **絶対ルール**: **数字が「小さい」方が「エラい（優先される）」**。
*   例えば、Rule A（優先度: 100, 許可/Allow） と Rule B（優先度: 200, 拒否/Deny）が設定されていた場合、「数字が小さい」100番のルールが勝つため、通信は許可されます。

## 3. 決して消せない「デフォルトの3大ルール」

新しいNSGを作成すると、優先度のリストの一番下（**`65000`, `65001`, `65050`** あたりの超巨大な数字）に、初期設定として3つのルールが最初から書き込まれています。

> [!WARNING]
> **🚨 Exam Trap (デフォルトルールの存在)**
> これら3つのルールは**「絶対に削除（ゴミ箱アイコン）することができません」**。
> どのようなルールが入っているか、試験のために暗記してください。
> 
> 1.  **[許可 / Allow] `AllowVnetInBound` / `AllowVnetOutBound`**
>     ➡️ 「同じVNet(LAN)の中にいる仲間同士の通信は、全部フリーパスで通してあげる」
> 2.  **[許可 / Allow] `AllowAzureLoadBalancerInBound`**
>     ➡️ 「Azureの公式ロードバランサーからの死活監視（生きてるか？の確認）通信だけは通してあげる」
> 3.  **[拒否 / Deny] 💀 `DenyAllInBound` (優先度 65500)**
>     ➡️ **【最強の盾】「上記2つに当てはまらない、インターネット等外部の知らない奴らからの通信は "すべて" 拒否して捨てる」**
> 
> つまり、新しくWebサーバーのVMを作っても、**「優先順位100番でPort80を許可する(Allow)」というルールを自分で手動で追加してあげない限り、最下層に鎮座する『DenyAllInBound』の盾に跳ね返されて、世界中の誰からもWebサイトを見てもらえない（アクセスできない）**状態になっています。

試験で「外部からアクセスできない。なぜか？」と問われたら、正解は「優先度の高いAllowルールを追加していないから」です！
