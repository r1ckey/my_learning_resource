# Azure Week 4②：監視と復旧 (Monitor & Backup)

インフラ編（Part 1〜2）の最後を飾るのが、組み上がったシステムを「監視（Monitor）」し、もしランサムウェア等で破壊されたら「復旧（Restore）」に走る運用保守アーキテクチャの完成です。

ここについて深く学ぶことは、のちの「データエンジニアリング（Part 3〜4）」で発生するエラーログをクエリ（KQL）で解析していくための超重要な布石になります。

## 1. アラートの起点「Azure Monitor」

Azure内の全てのリソース（VM、SQL、Databricks等）から「心拍数（メトリクス）」と「日記帳（ログ）」を自動的にかき集める巨大な集音マイクが **Azure Monitor** です。

*   **メトリクス (Metrics)**: 「CPU使用率 85%」「ネットワーク送信量 100MB」といった【数字データ】。
*   **ログ (Logs)**: 「14:00にAさんがログインに失敗した (Error 401)」といった【文字データ】。

### 📡 アラートルールとアクショングループ
「VMのCPUが90%を超えた！どうしよう！」を自動化します。
1.  **Alert Rules (アラートルール):** 「CPUが90%を5分間超えたら…」という引き金。
2.  **Action Groups (アクショングループ):** 上記の引き金が引かれたら、**「ITチームにメールを送信し、Slack(Webhook)に通知し、さらに自動で再起動用のPythonスクリプト(Functions)を叩く」** という『一連の動作の束』です。

## 2. ログの大海原「Log Analytics Workspace」

> [!IMPORTANT]
> **🚀 Exam Focus & Data Engineerへの架け橋**
> 各サーバーのテキストログは、そのまま置いても誰も読めません。これらを一箇所にバケツリレーのように集約して、表（データベース）の形に貯めていく巨大な貯水池が **Log Analytics Workspace (ログアナリティクス・ワークスペース)** です。
> 
> ここに溜まった膨大なログを、データエンジニアは **『KQL (Kusto Query Language)』** というSQLに似た強力な言語を使って、「エラーを出している犯人のIPアドレス」などを超高速で抽出・分析します。
> （KQLの記法： `AzureActivity | where Level == "Error" | sort by TimeGenerated desc` のようにパイプ `|` で繋ぎます）。

## 3. Azure Backup (Recovery Services Vault の魔法)

「マルウェア（身代金要求ウイルス）に感染して、VMのファイルサーバー上のデータが全部『暗号化』されてしまった！しかもハッカーご丁寧にサーバーの中でファイルを『削除』までしてしまった！」
こういう絶望的な状況を救うのが、PaaSのバックアップ機能です。

### Recovery Services Vault (RSV / 復旧サービスのコンテナ)
バックアップデータを保管しておく強固な金庫です。

> [!WARNING]
> **🚨 Exam Trap (消されたVMのバックアップはどうなる？)**
> これがAZ-104試験で最も意地悪な問題です。
> 
> **[問題]**: 「悪意のある人間がOwner権限を乗っ取り、**VM自体を『削除（Delete）』**してしまいました。VMが消滅した今、Recovery Services Vault (金庫) の中に残っていた『そのVMの中身のバックアップデータ』も道連れに消えてしまうのでしょうか？」
> 
> **[絶対の正解]**: **いいえ。バックアップデータは完全に無傷でVault（金庫）の中に残り続けます。**
> 
> むしろ、この金庫の中に残っているバックアップデータを利用して、「一昨日（感染前）の健康な状態のVM」として、**「新しいVMをゼロから再構築（復元：Restore）する」**ことが可能です。バックアップが本体の削除と切り離されている（隔離されている）のがPaaSバックアップの最強の強みです。
