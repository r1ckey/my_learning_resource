# Core Week 1：データ基盤の進化とパラダイム（DWH / Data Lake / Lakehouse）

システム開発において、データを保存する「箱」の進化は、そのまま人類の計算能力の進化の歴史です。
それぞれの「箱」が『何を解決するために生まれ』、『なぜ死んで（限界を迎えて）次の手段が生まれたのか』を説明できなければ、アーキテクトとは呼べません。

## 1. 原理原則：OLTP vs OLAP の戦い

### 🛒 OLTP (Online Transaction Processing)
*   **代表格:** MySQL, PostgreSQL, Oracle DB, SQL Server
*   **目的:** Amazonでボタンを押した瞬間に「在庫を1減らして、売上を1増やす」。
*   **特徴:** 1件〜数件のデータを超高速で書き換える（更新・削除・挿入）ことに特化しています。「現在の正確な状態」を保つためのシステム。
*   **限界:** 「過去10年間の全国の売上平均と、気温の相関を計算して」という『広範囲の読み取り（集計）』のSQLを投げると、数時間フリーズして、最悪の場合ECサイト自体が落ちます。

### 📊 OLAP (Online Analytical Processing)
*   **目的:** 上記の「ECサイトが落ちる」のを防ぐため、真夜中にOLTPからデータを別の巨大なデータベースに**『コピーして隔離』**し、そこで集計や分析を思いっきり叩かせる仕組み。これが、のちのDWHの基礎です。

## 2. 三世代のデータ基盤アーキテクチャ

### 第一世代：データウェアハウス（Data Warehouse: DWH）
*   **誕生理由:** 散らばったOLTPのデータを1箇所の巨大なSQLサーバーに集約し、経営ダッシュボード用に見やすくする（OLAP）ため。TeradataやSnowflakeなど。
*   **特徴:** **スキーマ・オン・ライト（Schema-on-Write）**。データを入れる前にテーブルの「型（日付、文字など）」をカッチリ決めないと保存できません。
*   **限界の到来（死の理由）:** AI・機械学習のブーム到来。「画像」「音声」「ログのテキスト」などの**非構造化データ**を保存したくても、DWHは『綺麗なExcelの表（構造化）』しか受け付けないため、AIエンジニアから「使えない」と見放されました。

### 第二世代：データレイク（Data Lake）
*   **誕生理由:** DWHで見捨てられた「画像、音声、JSON」など、ありとあらゆる生データを「とりあえずそのままのぐちゃぐちゃの形（沼）」に全部ぶち込める巨大で安いストレージが必要になったため。Amazon S3やAzure ADLS Gen2。
*   **特徴:** **スキーマ・オン・リード（Schema-on-Read）**。保存する時はルール無用で何でも入る。読み出す時（クエリする瞬間）に初めてAIツールなどが形を解釈する。
*   **限界の到来（沼の腐敗）:** データの更新（UPDATE）や削除（DELETE）の仕組みが欠落していたため、間違って入ったゴミデータが永遠に残り、誰にも管理されない**「データのゴミ捨て場（Data Swamp: 沼が濁って腐った状態）」**へと化しました。

### 第三世代の到達点：データレイクハウス (Data Lakehouse)
*   **誕生理由:** **「データレイクの安さと無限の自由（非構造化データ対応）」** と **「DWHの堅牢さ（UPDATE/DELETEの信頼性、ACIDトランザクション）」** を、両方1つの場所で完璧に叶える究極の実装ができなかったのか？
*   **解決策:** それを実現したのが我らが **『Databricks』と『Delta Lake（オープンソースのテーブルフォーマット）』**です。安いADLSのストレージの上に「Delta」という魔法の監視層を敷くことで、ゴミ捨て場だった湖に、完全なSQLトランザクションをもたらしました。

> [!IMPORTANT]
> **🚀 Interview Insight (面接官の質問)**
> 「なぜあなたの会社はSnowflake（DWH）ではなく、Databricks（Lakehouse）を選んだのですか？」
> 
> *   **完璧な回答**: 「数年後の機械学習（ML）とAIの統合を見据えたためです。DWHはBI（過去の集計）には最強ですが、AIモデルを回すための『非構造化データの結合』や『Pythonの分散処理』において致命的に弱いです。レイクハウスアーキテクチャならば、BIの集計とAIの予測モデルを、全く同じデータソース・同じ権限（Unity Catalog）で動かすことができ、データコピーの金銭的・時間的コストをゼロにできるからです。」
